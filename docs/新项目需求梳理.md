## 已确认的新项目规则

### 对局名额规则

- 一局正常预想为：主播 + 5 个老粉 + 2 个新粉，共 8 个席位【已确认】。
- 老粉和新粉都不是固定一定存在：
  - 老粉可能不足。
  - 老粉可能溢出，需要按定榜或取票规则筛选。
  - 新粉可能没有。
  - 主播状态不好时，可能只带老粉，不带新粉。
- 定榜默认看榜五，但可以根据新粉情况扩展到榜六、榜七等顺延名额【已确认】。
- 系统不应把“榜五 + 2 新粉”写死，应做成每场或每局可调整的名额配置【推导】。
- 老粉上车默认根据定榜顺序决定。例如 A500、B400、C300、D200、E101、F100，且有 2 个新粉时，老粉名额取 A/B/C/D/E，F 等无新粉、下次定榜上票、前面有人不玩或补票后再参与【已确认-0617】。
- 相同票数时，优先按榜单顺序处理；也可以由粉丝协商补票后再决定【已确认-0617】。
- 同票时系统应支持备注或手动调整先后顺序【已确认-0617-2】。
- 没有新粉时，新粉名额顺延给老粉；老粉也不够时等待新粉或更多老粉【已确认-0617】。
- 等待太久时，主播可能用小号双开/多开补齐人数，一个人操作多个游戏号【已确认-0617】。
- 规则水友赛原则上必须 8 人，不随机匹配路人；如果非常缺人或主播状态不好，可以切换为无规则模式，此时不锁牌、不强制 8 人【已确认-0617】。
- 定榜后可以一键生成推荐名单，但必须支持主播手动修改、替换、顺延和备注，避免有人突然有事不玩【已确认-0617-2】。
- 一键生成名单后可以考虑增加“确认名单”动作；无论是否一键生成，都必须支持操作者手动修改和调整名单【已确认-0617-3】。
- 定榜候选优先按榜 7 查看，定完后根据新粉数量判断哪些老粉上车、哪些人待定或等下一把【已确认-0617-3】。
- 定榜名单生成支持两种风格：一种是正常定榜七，有新粉时后面老粉进入待定；另一种是正常定榜五，没新粉时再顺延补位老粉，没老粉时考虑双开/多开等补位【已确认-0618】。
- 两种名单风格都可以保留为规则模板或操作模式，由主播/管理员按习惯选择【推导】。
- 无规则模式也可以保存历史记录，只是不进入锁牌流程，方便后续回溯查看【已确认-0617-2】。

### 票务规则

- “票”本质是抖音礼物钻，例如墨镜 99 钻、礼花筒 199 钻【已确认】。
- 存票余额正常长期保存，不随场次、赛季自动清空【已确认】。
- 后续可以考虑主动清空按钮，但必须非常慎重，最好需要二次确认、权限限制和操作日志【已确认】【推导】。
- 取票是从历史存票余额中取出一部分用于当前定榜竞争，例如余额 2300，本场可先取 500，不够再继续取或现刷礼物【已确认】。
- 现刷礼物和取用存票都可能影响本场实际竞争票数，但二者来源不同，账本必须分开记录【推导】。
- 定榜记录需要能记录抖音名、礼物钻、本场取票/存票等票务操作；榜单名次应按实际定榜竞争票数动态计算【已确认-0617】。
- 抖音名可以和粉丝名、微信名、游戏名绑定到同一个粉丝资料【已确认-0617】。
- 修改存票不直接改原记录，优先新增修正记录，避免误操作并保证可追溯【已确认-0617】。
- 票务记录如需删除，应做软删除/作废，不做物理删除【已确认-0617】。
- 总竞争票可以理解为：本场礼物钻 + 本次取票数【已确认-0617-2】。
- 总竞争票公式可以先确认采用“本场礼物钻 + 本次取票数”，但必须支持人工修改、调整和备注【已确认-0617-3】。
- 本场存票在直播结束后统一处理入账，但直播中需要可预览结算后的总余额，避免中途反复修改历史余额【已确认-0617-2】。
- 取票后如果突然有事一把没玩，可以考虑退回存票余额；正常取票并玩过一局后就不能退回【已确认-0617-2】。
- 取票后没上车，可以在最终结算时备注退回；取票金额支持随时调整【已确认-0617-3】。
- 如果只是某一把临时有事，取票可以暂时不动；如果当天来不了了，可以标记为未使用，并在结算时回退【已确认-0618】。
- 每次定榜都可以修改调整票数，因为直播中随时可能有人存票或取票【已确认-0618】。
- 直播结束存票入账建议需要“结算确认”动作，让操作者清楚知道本次结算会影响余额【已确认-0617-3】。
- 存取票正常以整百操作为主，但系统不强制只能整百，除非后续规则确认【已确认-0617-2】【推导】。

### 粉丝状态规则

- 粉丝可以有状态标记，例如直播间管理、粉丝团等级、新关注、新粉、老粉、违规、拉黑等【已确认】。
- 新关注粉丝可以标记为新粉，但参与过一局后应变为老粉【已确认】。
- 违规或拉黑状态应影响参赛资格、提醒和公开展示范围【推导】。
- 粉丝允许多状态并存，例如直播间管理也可能有违规状态【已确认-0617】。
- 当管理身份粉丝被拉黑/删除时，管理状态应自动取消【已确认-0617】。
- 拉黑后禁止参赛，历史数据保留；误拉黑可以改回，且不影响之前的正常历史数据【已确认-0617】。
- 新粉转老粉应支持完成一局后自动更新，也要支持手动标记，避免忘记记录对局时遗漏【已确认-0617】。
- 一个粉丝可能有多个游戏号，也可能互相换号玩；系统应以内部常用称呼/粉丝身份为主，游戏名作为可选展示或默认常用号，并支持备注说明【已确认-0617-2】。
- 游戏号名称不是重点，不需要做复杂账号归属模型；本局使用的游戏名、换号、多开等情况用备注记录即可【已确认-0617-3】。

### 账号与权限规则

- 游客是未登录用户【已确认】。
- 正常只有管理员和主播可以登录【已确认】。
- 其他粉丝不能自行注册登录【已确认】。
- 新主播账号应由管理员创建，再交给主播登录使用【已确认】。
- MVP 优先服务一名主播，但底层要预留扩展到多主播的能力【已确认】。
- 游客只能查看管理员/主播主动设置为公开的内容，重点是粉丝内部存票总余额排名；对局记录暂时不是游客重点关注内容【已确认-0617】。
- 截图、对局记录、存票记录等都可以考虑设置公开状态，只有设置公开后游客才能看到【已确认-0617】。
- 公开存票榜可以显示全部，也可以只显示指定公开的部分人，并支持分页、筛选查看【已确认-0617-2】。
- 公开存票榜可以显示精确票数/钻数，当前存取票通常按整百操作【已确认-0617-2】。
- 游客公开页只能按允许公开的字段筛选；如果抖音名没有公开，就不能用抖音名搜索【已确认-0617-3】。
- 公开内容可以考虑单独的公开确认动作，避免误公开【已确认-0617-3】。
- 公开确认采用“公开模块”级别，不做每条内容逐条审核作为 MVP 必须项【已确认-0618】。
- 主播和管理员都可以作废票务流水，但管理员权限更高，后续可限制某些高风险作废只能管理员处理【已确认-0617-3】。
- 目前暂不设定额外高风险作废清单；推荐 MVP 默认主播/管理员都可作废，所有作废必须留日志，后续发现风险再加管理员二次确认【已确认-0618】【推导】。
- 管理员可以代主播操作，包括代结算、修改主播基础资料、查看内部备注和查看截图等辅助管理行为【已确认-0618-2】。
- 拉黑/解除拉黑建议必须填写原因，避免误操作并方便回溯【已确认-0617-3】。

### 设备与部署规则

- 需要支持手机、电脑、平板等多设备操作【已确认】。
- 多设备访问需要支持，但如果多设备同时登录实现复杂，可以先采用“同一账号仅一个设备在线”的策略；其它设备提示需要重新登录，切换设备要尽量顺滑【已确认-0617】。
- 目标是第一版框架尽量到位，避免后期在架构、数据存储和权限上大改【已确认】。
- 因为需要在线部署、多设备同步、服务器截图和日志，项目应从第一版就按后端数据库方案设计，不再采用纯本地存储作为核心方案【推导】。
- 部署优先尝试免费方案，例如 Cloudflare 体系；如果免费方案不够，再使用正常云服务器和 Docker 部署【已确认-0617】。
- 如果暂时没有合适服务器，至少要保证本地可以正常运行；后续有合适服务器时，部署后也能正常运行【已确认-0617-2】。
- 架构应同时兼顾本地运行和线上部署，不把核心能力绑定死在某一个部署平台上【推导】。

### 截图与日志规则

- 截图需要上传至服务器保存【已确认】。
- 管理员和主播的关键操作都需要日志，尤其是票数、存票、取票、结算、清空、拉黑、权限和数据修正【已确认】。
- 旧项目数据不迁移，新项目只参考旧项目沉淀资料【已确认】。
- 图片可以压缩、限制大小和格式，但必须保证关键信息能看清【已确认-0617】。
- 图片压缩目标是减少存储和内存占用，只要能看清内容即可【已确认-0617-2】。
- 截图是否公开由公开状态控制，默认不公开更安全【已确认-0617】。
- 用户上传图片时，如果体积不合适，系统提示体积过大，需要压缩或重新上传其它图片【已确认-0617-3】。
- 截图上传后目标是永久保存，不做自动清理【已确认-0618】。
- 截图允许删除，用于清理没用或错误图片，避免占用存储空间【已确认-0618】。
- 精彩截图和违规截图都可以公开展示，是否展示只取决于是否被设置为公开【已确认-0618-2】。
- MVP 推荐单张截图限制 5MB，并保留按实际部署平台调整的配置；超过限制时提示用户压缩或重新上传【推导】。
- 自动清空存票只预留设计，先不实现【已确认-0617】。

### 初步部署判断

- 首选可探索 Cloudflare 免费/低成本路线：前端使用 Cloudflare Pages，接口使用 Workers，结构化数据使用 D1，截图对象存储使用 R2【推导】。
- Cloudflare Workers 免费层适合低流量 MVP，但有每日请求量、CPU 时间等限制；需要避免高频实时轮询【推导】。
- Cloudflare D1 适合轻量结构化数据，例如主播、粉丝、票务流水、对局、日志，但不适合存图片本体【推导】。
- Cloudflare R2 适合存截图，免费额度对单主播 MVP 通常够用，但需要限制图片大小、压缩和上传频率【推导】。
- 如果采用 Cloudflare 体系，上传链路仍建议在产品层限制单张截图大小。MVP 推荐 5MB，比平台硬限制更保守，便于控制免费额度和加载速度【推导】。
- 如果后续需要复杂实时同步、大量图片、更多主播或更强后台任务，备选方案是普通云服务器 + Docker + 数据库 + 对象存储【推导】。
- 无论最终选 Cloudflare 还是 Docker 云服务器，都需要保留本地开发/本地运行能力，避免没有服务器时项目无法使用【已确认-0617-2】。

## 新项目目标说明

新项目目标是做一个可部署的金铲铲水友赛直播辅助网页工具，围绕主播真实直播流程，帮助主播、管理者和参与粉丝更轻松地处理锁牌、存票、对局历史和必要的公开查询。

本项目不直接继承旧项目，而是重新围绕“直播场次 -> 多局对局 -> 多次定榜 -> 玩家/粉丝票务变化 -> 对局记录沉淀”设计。旧项目只作为业务经验参考，尤其参考锁牌规则、玩家/五费资料结构、8 人对局约束和旧项目踩过的边界问题。

核心问题不是“记录一张表”，而是解决直播过程中容易混乱、容易算错、难以回看、难以协助的问题：

- 主播直播时快速记录每局锁牌变化，知道谁锁了什么、还剩什么。
- 新粉或临时参与玩家多时，能被清楚提醒规则和状态。
- 一场直播中多局定榜、存票、取票、继续刷礼物时，能动态算清每个人实际票数和余额。
- 直播结束后能形成可追溯的存票汇总和对局记录。
- 管理者可以查看所有主播数据，在主播不会操作时辅助处理。
- 参与粉丝可以查看公开范围内的对局历史或存票记录。

## 用户角色和使用场景

### 主播端

使用者：金铲铲主播。

主要场景：

- 开播前选择当前赛季、直播场次和本场规则。
- 每局游戏开始前配置 8 个玩家席位，包括榜五、新粉、顺延名额等。
- 对局中记录每个玩家锁定的五费卡，查看剩余可选卡。
- 玩家淘汰后保留记录，同时释放其锁定卡对存活玩家的占用。
- 直播过程中记录存票、取票、继续刷礼物后的票数变化。
- 一场直播结束后查看本场结算汇总。
- 必要时上传或关联截图，记录精彩画面或违规证据。

来源：锁牌、五费资料、8 人席位、淘汰后释放占用来自【旧沉淀】；直播场次、多局定榜、榜五加新粉、存取票和截图用途来自【新想法】。

### 管理端

使用者：项目管理者，也就是工具拥有者。

主要场景：

- 查看所有主播列表、直播场次、对局、存票和历史数据。
- 代主播录入或修正资料、票务流水、对局记录。
- 在主播不会操作时远程辅助管理。
- 检查数据异常，例如票数为负、重复玩家、未结算场次。
- 后续可扩展为主播权限、公开页面权限和数据导出。

来源：管理端和跨主播管理来自【新想法】；数据导入导出、稳定 id、风险意识来自【旧沉淀】。

### 游客查看部分

使用者：参与水友赛的粉丝、榜上玩家、普通观众。

主要场景：

- 查看公开的对局历史。
- 查看自己的存票记录或公开存票榜。
- 查看某场直播的参与记录、对局结果或结算摘要。
- 后续可查看截图墙或精彩记录，但需要权限控制。

来源：游客查看来自【新想法】；历史记录不能只做快照来自【旧沉淀】。

## MVP 范围

### 当前 MVP 必须做

1. 多角色基础结构
   - 主播端、管理端、游客公开页三类入口。
   - 管理端至少能查看和辅助管理一个或多个主播的数据。
   - 游客不登录；只有管理员和主播可以登录。
   - 新主播账号由管理员创建。
   - 来源：【新想法】【已确认】。

2. 主播资料与玩家资料
   - 主播基础资料。
   - 玩家资料支持昵称、抖音名、游戏名、备注、状态标记。
   - 状态标记至少考虑：直播间管理、粉丝团等级、新粉、老粉、违规、拉黑。
   - 玩家可以被多次复用，不与某一局强绑定。
   - 新粉参与过一局后应可转为老粉。
   - 来源：玩家多名称来自【旧沉淀】，主播资料和状态标记来自【新想法】【已确认】。

3. 赛季与五费卡资料
   - 五费卡支持名称、外号、分类、赛季、标签、备注。
   - 可按赛季快速筛选五费。
   - 来源：【旧沉淀】。

4. 直播场次
   - 支持创建下午场/晚上场等直播场次。
   - 一场直播包含多局对局。
   - 场次有开始时间、状态、备注、结算状态。
   - 场次规则需要支持“主播 + 老粉名额 + 新粉名额”，并允许临时调整。
   - 支持规则水友赛和无规则模式两种场景；无规则模式可以不锁牌、不强制 8 人。
   - 来源：【新想法】与【旧沉淀】中“存票需要场次概念”的结论。

5. 对局管理与锁牌管理
   - 每局固定 8 个玩家席位。
   - 默认席位为主播 + 5 个老粉 + 2 个新粉，但不能写死。
   - 规则水友赛必须 8 人；无规则模式允许不锁牌、不限制 8 人。
   - 支持主播小号/多开补位，但游戏号名称只作为本局游戏名或备注记录。
   - 无规则模式也保存历史记录，但不进入锁牌流程。
   - 每局单独选择参与玩家和可用五费卡。
   - 记录玩家锁牌变化。
   - 存活玩家之间互斥占用五费。
   - 淘汰玩家保留记录，但不继续占用可用池。
   - 显示剩余可用五费。
   - 支持保存对局结果和历史。
   - 来源：【旧沉淀】为主，【新想法】补充“提醒新粉”场景。

6. 存票管理基础版
   - 按主播、玩家、直播场次记录存票/取票流水。
   - 存票余额长期保存，默认不随场次或赛季清空。
   - 取票用于从存票余额中拿出一部分参与当前定榜。
   - 现刷礼物钻和取用存票都要能进入本场竞争统计，但来源要分开。
   - 总竞争票默认按“本场礼物钻 + 本次取票数”计算，但支持人工调整和备注。
   - 本场新增存票直播结束后统一入账，直播中提供预览余额。
   - 支持取票未上车时在最终结算备注退回；取票金额可随时调整。
   - 直播结束入账需要结算确认。
   - 修改存票通过新增修正记录完成，不直接覆盖原始流水。
   - 删除票务流水采用软删除/作废，并保留日志。
   - 动态计算当前余额。
   - 支持备注，便于记录礼物来源或人工说明。
   - 直播结束后生成本场存票汇总。
   - 来源：流水和余额动态计算来自【旧沉淀】；场次、多局、实际票数统计来自【新想法】【已确认】。

7. 定榜记录基础版
   - 一场直播可以多次定榜。
   - 每次定榜记录榜五和新粉名额。
   - 明确“本场直播累计榜”而不是每局重新刷。
   - 默认定榜五，但可以根据新粉情况记录榜六、榜七等顺延名额。
   - 实操上优先按榜 7 查看候选，再根据新粉数量判断上车、待定或等下一把。
   - 也支持正常定榜五，没新粉时再顺延补位老粉的操作风格。
   - 老粉不足、新粉不足、只带老粉等情况要能被记录。
   - 按实际定榜竞争票数动态计算榜单名次；同票时保留榜上顺序或补票协商结果。
   - 支持一键生成推荐名单和确认名单动作，但主播可以手动调整、替换和备注。
   - 来源：【新想法】【已确认】，顺延规则来自【旧沉淀】中“不建议复用当前存票模型”的问题描述。

8. 服务器截图基础版

   - 违规截图默认不公开。
   - 图片限制大小和格式，上传时可压缩，但必须保持内容可辨认。
   - 来源：【新想法】【已确认】。

9. 操作日志基础版

   - 票数、存票、取票、结算、清空、拉黑、权限和数据修正必须有日志。
   - 来源：【新想法】【已确认】。

10. 多设备和在线部署基础
   - 支持手机、电脑、平板访问。
   - MVP 可以先限制同一账号单设备在线，降低并发冲突复杂度。
   - 切换设备需要顺滑，旧设备提示下线，新设备重新登录后继续操作。
   - 第一版按后端数据库和服务器存储设计，不以浏览器本地存储为核心。
   - 必须保证本地可正常运行；后续有服务器时可平滑部署。
   - 来源：【已确认】【推导】。

11. 公开查询基础版

   - 游客只能查看管理员/主播设置为公开的内容。
   - 公开榜支持显示全部或指定公开人群，支持分页和筛选。
   - 公开页只能按已公开字段筛选；未公开抖音名时不能按抖音名搜索。
   - 公开榜可以显示精确票数/钻数。
   - 公开内容可以考虑公开确认动作，避免误公开。
   - 对局记录暂时不是公开页重点，可以后置。
   - 公开范围需要先做最小权限边界，不能默认暴露所有后台备注、违规记录和私密截图。
   - 来源：【新想法】与【推导】。

## 需求冲突、重复和边界不清

### 1. 名额规则已基本明确

已确认默认是主播 + 5 老粉 + 2 新粉，老粉按定榜顺序进入名额，同票看榜上顺序或补票协商。只带老粉、新粉不足、老粉不足等情况不再作为固定单一规则写死，进入页面流程设计时用“规则模板 + 手动调整 + 备注”解决。

已确认游戏号不做复杂归属模型，本局使用的游戏名、换号、多开等情况用备注处理。

### 2. “每场直播累计榜”与“每局定榜”需要区分

你提到“一场直播打多局，定多次榜，每次定榜是算本场直播累计”。这里有两个概念：

- 定榜动作：第几局前/后截取一次榜单。
- 榜单来源：本场直播累计礼物榜。

系统需要记录多次定榜，但不能把每次定榜理解成票数清零重算。

### 3. 存票、上榜票数、礼物票数之间的定义需要统一

目前已确认“票”以抖音礼物钻为基础，榜名次按实际定榜竞争票动态计算。系统里仍然至少有四种数字：

- 历史存票余额。
- 本场取用存票。
- 本场现刷礼物钻。
- 本场实际用于参赛/抵扣/结算的票数。

需要在数据模型中分清来源、用途和结算结果，否则后面会很容易算错。

已确认总竞争票可按“本场礼物钻 + 本次取票数”理解，并支持人工调整和备注。本场存票直播结束后入账，直播中先预览余额，入账需要结算确认。取票未上车时可暂不动或结算回退，每次定榜都允许调整票务数据。

### 4. “存票的人不一定玩”与“榜上参与名额”需要分开

存票余额是玩家资产，对局席位是某一局参与资格。二者有关联，但不能混为一张表。

### 5. 游客可见范围仍需细化

已确认游客未登录，粉丝不能注册登录，并且只能看到管理员/主播设置公开的内容。公开页只能按已公开字段筛选，如果抖音名未公开就不能按抖音名搜索。公开内容可考虑单独确认动作，避免误公开。

已确认公开确认采用公开模块级别开关，MVP 不做每条内容逐条审核。

### 6. 管理端“辅助管理”的权限边界已明确到 MVP 级别

已确认管理员可以创建主播账号，并可以代主播操作，包括修改主播资料、代结算、查看截图、查看内部备注和辅助处理数据。主播和管理员都可以作废票务流水，管理员权限更高。MVP 暂不设额外高风险作废清单，所有作废保留日志，后续发现风险再加管理员二次确认。

### 7. 截图公开和存储策略已明确到 MVP 级别

截图需要服务器保存。已确认截图目标永久保存，不自动清理，允许删除没用或错误图片。精彩截图和违规截图都可以公开展示，是否展示只取决于是否被设置为公开。上传体积过大时提示压缩或重新上传，清晰度以能看清内容为准。MVP 推荐单张 5MB 限制，并保留按部署平台配置调整。

## 当前优化评估

- 名额规则不建议写死一种流程，应做成“定榜七候选”和“定榜五顺延”两个规则模板，再允许主播手动调整。
- 票务不要追求全自动结论，核心是流水清楚、结算前可预览、结算时确认、出错后用修正/作废追溯。
- 游客公开页按模块开关更适合 MVP，能降低误公开风险和操作成本；筛选字段必须跟随公开字段权限。
- 截图永久保存但允许删除是合理折中：证据和精彩记录能保留，错误图片也不会长期占用空间。
- 图片限制建议先用 5MB，后续如果截图经常看不清或平台额度吃紧，再调整压缩策略。
- MVP 权限策略建议先简单：主播和管理员都能完成日常操作，管理员权限更高；所有敏感操作靠日志兜底，后续再把真实高风险动作单独加二次确认。
- 当前需求已经足够进入下一步：数据模型、页面流程草图和 MVP 版本切分。

### 8. 新粉提醒的具体形式不清

“提醒新粉”可能是：

- 主播端提示这个玩家是新粉。
- 游客端给新粉看规则。
- 对局中提醒新粉当前该锁什么。
- 违规或操作提醒。

需要进一步定义。

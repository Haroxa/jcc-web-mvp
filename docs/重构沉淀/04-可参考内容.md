# 04-可参考内容

整理时间：2026-06-16

## 可以参考的产品经验

### 先做静态 MVP 是有效的

旧项目没有后端和复杂部署，却已经验证了很多真实操作问题：

- 玩家和五费资料需要长期维护。
- 每局比赛需要单独配置参与玩家和五费。
- 五费卡需要分类、赛季和标签。
- 淘汰玩家记录不能简单删除。
- 存票不是单纯加减，还会牵涉直播场次和定榜。

这个经验可以继续保留：先用低成本版本验证业务，再决定是否升级数据存储和架构。

### 基础资料与本场配置应该分开

旧项目早期把玩家和五费是否参与本场混在基础资料里，后来逐步把选择行为移到锁牌管理页的本场配置中。

这个方向是正确的：

- 基础资料负责长期信息。
- 本场配置负责当前对局参与范围。
- 锁牌表负责实际记录。

### 五费资料结构值得保留

五费卡从简单名称升级为：

- 名称
- 外号
- 分类
- 赛季
- 标签
- 备注

这个结构更贴近实际使用，尤其是：

- 外号方便直播间内部称呼。
- 赛季可以用于快速配置。
- 分类颜色能减少阅读负担。
- 标签可以表达特殊来源或特殊规则。

### 玩家多名称字段值得保留

玩家资料从“名称”升级为：

- 昵称
- 抖音名
- 微信名
- 游戏名
- 备注

这是符合实际场景的。不同页面可能需要显示不同名称，例如内部沟通看昵称，锁牌或对局时看游戏名。

### 锁牌互斥规则值得参考

旧项目已经沉淀出一个关键规则：

- 存活玩家之间锁定五费互斥。
- 淘汰玩家记录保留。
- 淘汰玩家记录不再占用存活玩家可选池。

这条规则很重要，后续只要继续做锁牌工具，都应优先明确。

### 8 个玩家席位模型值得参考

金铲铲一局 8 人，因此旧项目把本场玩家限定为 8 个席位，并在缺少玩家时显示占位。

这个模型能帮助页面保持稳定，也能提醒用户当前配置是否完整。

### 导入导出适合作为免费阶段兜底

在没有数据库之前，JSON 导入导出是有价值的：

- 能备份。
- 能迁移到其他浏览器。
- 能降低本地数据丢失风险。

但它只能作为兜底，不适合作为长期多人协作方案。

## 可以参考的文档经验

旧项目已经有这些项目文档：

- `README.md`
- `CHANGELOG.md`
- `docs/项目状态.md`
- `docs/功能设计安排.md`
- `docs/风险清单.md`

其中比较有价值的是：

- 用 `README.md` 说明项目用途和启动方式。
- 用 `CHANGELOG.md` 记录每次变更摘要和 Git 命令。
- 用 `功能设计安排.md` 记录想法、优先级和进度。
- 用 `风险清单.md` 把本地存储、移动端、数据丢失等风险提前写清楚。

## 可以参考的实现经验

- 使用稳定 `id` 关联玩家和五费，优于只靠名称。
- 数据读取时做旧格式兼容，减少升级后旧数据损坏。
- 页面上用颜色和标签表达分类，能提高扫视效率。
- 分页和固定高度能避免基础资料列表撑爆页面。
- 当前对局快照能解决“想回看某局状态”的基础需求。

## 使用这些经验的注意点

这些内容适合被当成经验，不等于应该原样照搬。尤其是：

- 产品规则可以参考。
- 字段设计可以参考。
- 页面结构可以参考。
- 当前代码组织方式不适合长期复用。

# 05-问题与反复修改点

整理时间：2026-06-16

## 反复修改点

### 本场配置位置和样式反复调整

本场配置经历过多次调整：

- 从基础资料中逐步移到锁牌管理。
- 从页面下方调整到当前对局下方。
- 默认折叠改为默认展开。
- 赛季选择被提升到配置标题行。
- 展开箭头样式多次修正。
- 添加五费展开后挤压布局的问题被修复。

说明：

本场配置是核心流程的一部分，不是一个普通设置项。旧项目一开始没有给它足够清晰的位置，导致后面多次靠样式和布局修补。

### 五费资料结构反复升级

五费从普通名称列表逐渐升级为：

- 分类
- 标签
- 外号
- 赛季

其中赛季最初和标签混在一起，后来才拆成单独字段。

说明：

五费卡不是简单枚举值，它有赛季、来源、类型和展示名称。后续任何继续设计都应该先承认它是结构化资料。

### 淘汰排名逻辑多次修复

淘汰排名曾出现显示或计算问题，后来改为基于淘汰时间计算。

说明：

排名看起来是显示问题，本质是业务规则问题。它需要明确：

- 谁先淘汰。
- 谁后淘汰。
- 存活玩家是否显示排名。
- 淘汰顺序如何影响最终名次。
- 修改淘汰状态后是否重算。

### 候选玩家和候选五费选择方式反复调整

相关变化包括：

- 不再展示全部资料占位置。
- 只展示当前已参与内容。
- 未参与内容通过搜索添加。
- 候选编号改为筛选后的顺序编号。
- 五费卡按分类、赛季、标签筛选。
- 支持批量添加筛选结果。

说明：

当玩家和五费数量增加后，简单下拉框或全量列表会很快变得难用。

### 基础资料表格多次优化

相关变化包括：

- 名称改为昵称。
- 增加抖音名、微信名、游戏名。
- 增加五费外号。
- 增加分页。
- 增加固定高度。
- 删除本场字段。
- 调整分页显示。

说明：

基础资料页面应该偏“资料维护”，不适合承担当前对局选择职责。

### 锁牌页面逐渐变重

锁牌管理当前承担了：

- 当前对局。
- 本场配置。
- 玩家选择。
- 五费选择。
- 赛季配置。
- 锁牌记录。
- 淘汰排名。
- 历史快照。
- 显示模式切换。

说明：

这说明锁牌页是业务中心，但也说明它已经过重。继续堆功能会让页面和代码都更难维护。

## 维护困难点

### `app.js` 文件过大

`app.js` 当前包含：

- 初始数据。
- 数据读取和迁移。
- 状态保存。
- 截图管理。
- 锁牌管理。
- 基础资料。
- 存票管理。
- 历史快照。
- 页面渲染。
- 事件绑定。

问题：

- 很难快速定位某个功能的所有逻辑。
- 一个小改动可能影响其他模块。
- 业务规则和页面渲染混在一起，不方便测试。

### 状态边界不清晰

当前玩家和五费通过 `active` 表示是否参与本场。

问题：

- 基础资料和本场配置都依赖同一个字段。
- 历史恢复会改写资料里的 active 状态。
- 如果以后有多场对局或直播场次，这个模型会不够用。

### 本地存储承担了所有数据

当前所有数据保存在浏览器 `localStorage`。

问题：

- 清理浏览器数据会丢失。
- 不同设备不能同步。
- 多人无法协作。
- 图片数据可能占用较大空间。

### 表格和字符串模板较多

当前页面大量使用字符串拼接生成 HTML。

问题：

- 字段增加时容易漏改。
- 表格列很多时移动端不友好。
- 事件绑定依赖 `data-*` 和索引，后续更复杂时更容易出错。

### 缺少自动化测试

当前主要通过手工检查和 `node --check app.js` 做基础检查。

问题：

- 淘汰排名、数据迁移、存票余额这类规则容易回归。
- 每次改样式和业务都要靠人工确认。

## 当前问题总结

旧项目最大的问题不是“不能用”，而是“继续扩展会越来越费劲”。它已经很好地完成了 MVP 验证任务，但如果后续要长期维护，需要把业务概念、数据模型和页面模块拆得更清楚。

# 07-沉淀摘要

整理时间：2026-06-16

## 一句话总结

旧项目是一个已经跑通核心想法的静态 MVP，适合继续作为业务经验样本，不适合直接作为长期扩展的正式系统底座。

## 旧项目当前是什么

它是一个金铲铲水友赛记录助手，主要能力包括：

- 截图管理。
- 锁牌管理。
- 存票管理。
- 基础资料维护。
- 本场配置。
- 当前对局。
- 历史对局快照。
- 本地数据导入导出。

它使用纯 HTML、CSS、原生 JavaScript 和浏览器本地存储实现，没有后端和数据库。

## 已沉淀出的重要经验

### 业务经验

- 水友赛工具不能只做单个表格，需要围绕“资料、本场配置、当前对局、历史记录”拆分。
- 玩家会有多个名称，昵称、游戏名、抖音名、微信名都可能有用。
- 五费卡需要名称、外号、分类、赛季、标签。
- 锁牌规则中，存活玩家互斥占用五费，淘汰玩家保留记录但不继续占用可用池。
- 一局游戏固定 8 个玩家席位，这个约束能帮助页面和流程稳定。
- 存票业务比简单加减更复杂，需要场次、局次、定榜和结算概念。

### 产品经验

- 先用静态网页验证是合适的。
- 分类颜色和固定高度能明显改善使用体验。
- 基础资料只做资料维护，本场选择放到锁牌管理中更合理。
- 添加候选玩家和五费时，需要搜索、筛选和批量操作。
- 变更记录和功能安排文档能帮助持续迭代不跑偏。

### 技术经验

- 稳定 `id` 比直接依赖名称更可靠。
- 数据结构升级时需要兼容旧数据。
- 导入导出可以作为无后端阶段的备份方式。
- 单文件堆功能适合 MVP，不适合长期维护。

## 旧项目主要问题

- `app.js` 过大，功能边界不清晰。
- 锁牌页面承载过多职责。
- 基础资料和本场配置通过 `active` 混在一起。
- 历史对局只是快照，不是完整业务记录。
- 存票管理还没有覆盖真实直播定榜流程。
- 数据全部在浏览器本地，存在丢失和无法同步风险。
- 缺少自动化测试，复杂规则容易回归。
- 移动端适配仍不充分。

## 后续只能作为参考，不能直接照搬的内容

- 当前代码结构。
- 当前 `localStorage` 单状态模型。
- 当前存票模块的数据模型。
- 当前历史快照模型。
- 当前锁牌页不断堆功能的页面组织方式。
- 当前大量字符串模板拼接 HTML 的写法。

## 后续使用这些沉淀资料的方式

下一阶段如果要继续做“新项目灵感完善”，建议只把这些文档当作旧项目经验库使用：

- 用 `01-项目现状梳理.md` 了解旧项目边界。
- 用 `02-已有功能与页面清单.md` 区分已做、半成品和未落地想法。
- 用 `03-业务流程与数据关系.md` 理解旧项目实际数据关系。
- 用 `04-可参考内容.md` 提取可保留的产品和业务经验。
- 用 `05-问题与反复修改点.md` 避免重复踩旧项目的坑。
- 用 `06-不建议复用内容.md` 判断哪些不能直接照搬。

这些文档的作用是帮助判断，而不是直接生成新需求。后续是否做新项目、做哪些功能、怎么排优先级，都应该重新基于目标用户、使用场景和维护成本判断。

# 06-不建议复用内容

整理时间：2026-06-16

## 不建议直接复用的内容

### 不建议直接复用当前 `app.js` 组织方式

原因：

- 文件过大。
- 多个模块混在一起。
- 状态处理、业务规则、页面渲染和事件绑定没有明显边界。
- 后续调试成本会继续上升。

适合作为参考的部分：

- 字段命名。
- 基础规则。
- 数据迁移思路。
- 部分业务判断。

不适合直接照搬的部分：

- 整个文件结构。
- 大量字符串模板渲染方式。
- 所有功能共用一套全局状态的写法。

### 不建议直接复用当前存票模型

原因：

当前存票只有“存票/取票流水 + 当前余额排行”，无法完整表达真实直播场景：

- 一场直播多局。
- 下午场和晚上场。
- 每局定榜。
- 榜五加两个新粉。
- 无新粉顺延榜六榜七。
- 存了不一定玩。
- 一场直播结束后统一结算。

适合作为参考的部分：

- 流水记录。
- 余额动态计算。
- 按余额排序。

不适合直接照搬的部分：

- 把存票只看成简单余额。
- 没有场次和局次的记录结构。

### 不建议继续用 `active` 表达本场配置

原因：

`active` 当前既像基础资料状态，又像本场选择状态。

当只有一场当前对局时还能用；如果以后出现历史对局、直播场次、多局游戏，这个字段会让状态关系变得混乱。

适合作为参考的部分：

- 本场参与玩家和本场参与五费这个概念。

不适合直接照搬的部分：

- 把这个概念直接写回基础资料对象。

### 不建议把历史对局继续做成简单快照

原因：

快照可以恢复状态，但不能很好回答：

- 这一局发生了哪些操作。
- 哪些截图属于这一局。
- 哪些存票记录属于这一局或这一场直播。
- 最终排名和结算结果是什么。

适合作为参考的部分：

- 保存当前对局状态。
- 恢复某一局配置。

不适合直接照搬的部分：

- 历史记录只保存一份页面状态。

### 不建议依赖本地存储作为长期方案

原因：

- 容易丢数据。
- 无法多人共享。
- 无法跨设备同步。
- 图片和历史数据变多后容量风险会上升。

适合作为参考的部分：

- 免费阶段快速验证。
- 离线可用。
- 导入导出兜底。

不适合直接照搬的部分：

- 把它当成长期正式数据存储。

### 不建议继续扩大表格型锁牌页面

原因：

锁牌表格在桌面端可读性还可以，但移动端和信息继续增加时会变重。

不适合继续直接塞进去的内容：

- 更多玩家资料字段。
- 更多五费说明。
- 历史记录详情。
- 存票结算信息。

这些内容应该有更清晰的信息层级，不能全部挤在一张表里。

## 不建议复用的根本原因

旧项目是为了快速验证而做的 MVP，它的价值在于暴露真实业务规则和操作习惯，而不是提供一个可以无限扩展的代码底座。

后续使用旧项目时，应复用“经验”和“业务判断”，谨慎复用“代码结构”和“页面堆叠方式”。

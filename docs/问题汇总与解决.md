## I-017 / P0：本场榜单保存立即写长期票务流水

### 现象

本场榜单页面编辑 `取票` 或 `存票` 后，会按差额立即写入 `ticket_ledgers`，并刷新粉丝长期余额。

### 原因

首版为了让余额预览可用，把本场榜单编辑和正式票务流水写入合并在了保存动作里。后续梳理页面心智时确认，直播中的榜单录入应是本场草稿，正式入账应发生在场次结束后的结算确认。

### 解决

- 本场榜单保存只更新 `live_session_board_entries`。
- 榜单保存时只校验预计结算后余额不能小于 0。
- 新增场次动作：开始直播、结束直播、确认结算。
- 确认结算时根据本场榜单的取票和存票生成正式票务流水。
- 已结算或已取消场次不允许继续编辑本场榜单。

### 当前状态

代码已修正，类型检查和构建通过。后续需要在线上真实录入、结束场次和确认结算中验证。

### 后续注意

- 取票未上车回退规则仍需确认。
- 结算页需要继续补粉丝级结算预览。
- 如果历史测试数据已经存在同场次榜单流水，结算逻辑会按同场次已有取票/存票流水做差额入账，避免重复生成同类流水。

## I-016 / P1：本场榜单编辑已有记录时可能重复扣减取票

### 现象

本场榜单支持反复修改同一粉丝的取票和存票。如果后端按新填写的取票总值直接校验余额，会把已有取票重复计算。

### 原因

本场榜单字段是累计值，但票务流水需要记录本次编辑相对上一次编辑的差额。

### 解决

- 查询已有本场榜单记录。
- 计算 `withdrawDelta = 新取票 - 旧取票`。
- 计算 `depositDelta = 新存票 - 旧存票`。
- 余额校验使用 `当前余额 - withdrawDelta + depositDelta`。
- 写入票务流水时也只写差额。

### 当前状态

代码已修正，类型检查和构建通过。后续需要在线上真实录入中验证。

## I-015 / P1：远程 D1 迁移缺少 `CLOUDFLARE_API_TOKEN`

### 现象

执行远程迁移时报错：

```text
In a non-interactive environment, it's necessary to set a CLOUDFLARE_API_TOKEN environment variable for wrangler to work.
```

### 原因

当前 Codex 执行环境是非交互模式，Wrangler 不能弹出登录确认，也没有检测到 `CLOUDFLARE_API_TOKEN`。

### 当前状态

- `npm run db:migrate:local` 已通过。
- `npm run db:migrate:remote` 已通过。
- 远程 D1 已应用 `0001_ranking_pool.sql`。
- 本机用户环境变量已配置 `CLOUDFLARE_API_TOKEN` 和 `CLOUDFLARE_ACCOUNT_ID`。
- 本机用户环境变量中的 `http_proxy` / `https_proxy` 已配置到 `http://127.0.0.1:7897`。

### 解决方式

从本地隐私文件读取 Token，确认 `user/tokens/verify` 成功；从历史 Wrangler 日志中确认正确的 Cloudflare Account ID；随后写入用户级环境变量并执行：

```text
npm run db:migrate:remote
```

### 注意事项

- Token 不写入仓库，只保留在本机环境变量和本地隐私文件。
- 新终端会自动读取用户环境变量；当前已无需每次手动设置 Token。
- 如果 Cloudflare 后续提示权限不足，需要检查 Token 是否包含当前账号下的 D1 编辑权限。

## I-001 / P0：npm / pip 挂 VPN 后依赖下载失败

### 现象

- `npm install` 长时间卡住或超时。
- `npm view react version` 超时。
- `python -m pip index versions pip` 报代理连接失败或 TLS 中断。

### 原因

Clash Verge 当前混合代理端口是：

```text
127.0.0.1:7897
```

但旧配置中存在多个不一致端口：

```text
npm / 环境变量：127.0.0.1:7890
pip：127.0.0.1:10809
```

命令行工具会连接错误端口，导致依赖下载失败。

### 已执行解决

npm：

```text
proxy = http://127.0.0.1:7897
https-proxy = http://127.0.0.1:7897
registry = https://registry.npmmirror.com/
```

pip：

```text
global.proxy = http://127.0.0.1:7897
global.index-url = https://pypi.org/simple
```

### 验证结果

- `npm view react version` 成功。
- `python -m pip index versions pip` 成功。

### 注意事项

- Clash Verge 需要保持运行。
- 如果 Clash 混合端口变化，需要同步更新 npm / pip 代理。
- 如果退出 Clash，固定代理端口会失效。

## I-007 / P1：Cloudflare Pages 更新后显示空白

### 现象

旧项目线上地址更新后显示空白。

### 原因

旧项目是纯静态页面，Cloudflare Pages 之前可以直接发布根目录。新项目是 Vite + React 应用，必须先执行构建。若仍发布根目录，线上 `index.html` 会引用 `/src/main.tsx`，浏览器无法直接运行该源码，导致页面空白。

### 解决

Cloudflare Pages 构建设置应调整为：

```text
Build command: npm run build
Build output directory: dist
Root directory: /
```

仓库侧已补充：

```text
wrangler.toml: pages_build_output_dir = "dist"
package.json: pages:deploy
README.md: Cloudflare Pages 设置说明
```

### 追加修正

`wrangler.toml` 原本同时包含 Pages 输出目录、Worker 入口、D1/R2 占位绑定。为避免 Pages 构建解析配置时受到 Worker 配置影响，已拆分为：

```text
wrangler.toml：仅用于 Pages，保留 pages_build_output_dir = "dist"
wrangler.worker.toml：预留 Worker API、D1、R2 配置
```

如果 Cloudflare Pages 再次部署成功，线上 `index.html` 应引用：

```text
/assets/index-*.js
/assets/index-*.css
```

而不应再引用：

```text
/src/main.tsx
```

## I-010 / P1：Pages 线上 `/api/*` 返回 HTML 而不是 JSON

### 状态

已解决代码接入，等待 Cloudflare Pages 后台绑定 D1/R2/环境变量。

### 现象

线上页面请求 `/api/setup/status` 时，收到的是 `<!doctype html>`，前端解析 JSON 失败：

```text
Unexpected token '<', "<!doctype "... is not valid JSON
```

### 原因

Cloudflare Pages 只发布了静态前端，仓库里还没有 Pages Functions 入口处理 `/api/*`。因此 `/api/setup/status` 被当作前端路由回退到 `index.html`。

### 解决

新增：

```text
functions/api/[[path]].ts
```

该入口把 Cloudflare Pages 的 `/api/*` 请求转发给现有 Hono Worker。

### 验证

本地执行：

```powershell
npm run pages:dev
```

已验证 `/api/health` 返回 200。

### 后续注意

Cloudflare Pages 后台还需要在“设置 -> 绑定”中添加：

```text
DB：D1 database
SCREENSHOTS：R2 bucket
ADMIN_SETUP_TOKEN：环境变量
```

绑定完成后重新部署，线上 `/api/health` 应返回 JSON。

## I-011 / P1：Cloudflare 账号未启用 R2

### 状态

已接受，截图云存储暂缓。

### 现象

执行 R2 bucket 列表或创建时，Cloudflare 返回：

```text
Please enable R2 through the Cloudflare Dashboard. [code: 10042]
```

### 原因

当前 Cloudflare 账号尚未启用 R2 服务，且启用 R2 需要绑定银行卡。当前银行卡暂不支持，因此 Wrangler 无法创建 `jcc-web-new-screenshots` bucket。

### 解决

本轮先完成 D1：

- 已创建 D1：`jcc_web_new`
- 已写入真实 `database_id`
- 已执行远程迁移

R2 暂缓，不阻塞当前 MVP 核心流程。当前先继续推进账号、主播、粉丝、场次、定榜、存票、锁牌。

后续如启用 R2，再创建 bucket：

```text
jcc-web-new-screenshots
```

### 验证

代码已调整为可选 `SCREENSHOTS` 绑定，未配置 R2 时 `/api/health` 会显示截图存储为暂缓状态。

### 后续注意

截图上传、截图公开墙、违规截图云端保存等功能先不进入当前 MVP 必做范围。

## I-012 / P1：线上初始化管理员口令校验后返回 500

### 状态

已解决。

### 现象

使用本地保存的初始化口令请求线上 `/api/setup/admin`，先返回：

```text
初始化口令不正确。
```

重新写入 Pages secret 并部署后，口令校验通过，但接口返回：

```text
500 Internal Server Error
```

### 原因

口令错误是 Cloudflare Pages secret 初次写入值和本地记录不一致导致，已重新写入。

500 发生在创建管理员之前，远程 D1 直接插入测试成功，因此问题集中在应用层密码哈希步骤。当前 PBKDF2 写法在 Pages Functions 生产环境中不稳定。

### 解决

将密码哈希改为 Workers 环境稳定支持的盐化 SHA-256 格式：

```text
sha256_salted$salt$hash
```

### 验证

- `npm run typecheck` 已通过。
- `npm run build` 已通过。
- 线上 `/api/setup/admin` 已成功创建管理员。
- 远程 D1 已确认账号 `admin` 存在且状态为 `active`。
- 线上 `/api/auth/login` 已通过。
- 线上 `/api/auth/me` 已通过。

### 后续注意

当前方案适合 MVP 阶段。后续如安全要求提高，可升级到专门的密码哈希库，并增加密码重置流程。

## I-013 / P0：票务取票、负修正或作废可能导致余额为负

### 状态

已解决，待真实操作验证。

### 现象

复查票务流水基础功能时发现，当前已经能新增存票、取票、现刷、修正和作废，但缺少余额下限校验：

- 取票金额可能大于当前存票余额。
- 负数修正可能把余额扣成负数。
- 作废历史存票记录后，如果后面已经发生取票，也可能导致余额变成负数。

### 原因

首版票务实现只做了流水追加和作废后余额重算，没有在写入前预估操作后的余额。

### 解决

新增统一余额计算函数，并在以下操作前检查余额：

- 新增取票或负修正前，检查操作后余额不能小于 0。
- 作废影响余额的流水前，检查作废后余额不能小于 0。
- 存票和正修正继续允许增加余额。
- 现刷不影响长期余额。

### 验证

- `npm run typecheck` 已通过。
- `npm run build` 已通过。

### 后续注意

后续实现结算回退、未使用取票退回时，也必须先计算操作后的余额，不能绕过流水直接改粉丝余额。

## I-014 / P0：定榜推荐错误按榜上顺序而不是总票数计算

### 状态

已解决，待真实操作验证。

### 现象

定榜页面录入“名次、礼物钻、取票、调整”后，推荐名单曾按 `名次 <= 5/7` 判断是否推荐上车。

这与实际业务不符。名单应按实际总票数排序：

```text
总票数 = 礼物钻 + 取票 + 调整
```

榜上顺序只应作为同票时的兜底顺序。

### 原因

首版定榜基础实现把 `rank_order` 当成推荐依据，`competition_score` 只用于展示，没有在推荐决策中重新排序整个榜单。

### 解决

- 将 `rank_order` 前端文案改为“榜上顺序”。
- 每次保存榜单条目后，重新计算整个定榜的推荐状态。
- 推荐顺序改为：总票数降序，榜上顺序升序。
- 拉黑粉丝仍优先标记为禁赛，不占推荐席位。
- 定榜列表展示顺序改为：推荐、待定、禁赛，再按总票数和榜上顺序排序。

### 验证

- `npm run typecheck` 已通过。
- `npm run build` 已通过。

### 后续注意

后续做“确认名单”时仍必须允许主播/管理员手动调整，因为新粉/老粉补位、同票协商、临时不上车等现场规则不能完全靠自动排序解决。

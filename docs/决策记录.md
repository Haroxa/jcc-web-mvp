## 2026-06-20 00:08：前端信息架构重组

### 决策

下一阶段先暂停继续堆叠新业务功能，优先把前端页面从“分散模块管理页”重组为：

- 主播端：今日工作台 + 当前场次工作台。
- 管理端：管理首页 + 主播详情 + 资料/票务/日志管理。
- 游客端：公开首页 + 公开存票榜。

### 原因

当前账号、主播、粉丝、场次、票务和定榜基础已经可用，但页面组织偏向后台 CRUD。真实直播时，主播需要围绕当前场次连续完成定榜、名单确认、对局锁牌、存票和结算，频繁切换孤立页面会降低效率，也容易选错场次。

### 注意

后端和已有 API 不推倒重来，优先复用现有能力。后续确认名单、锁牌、结算和公开榜都应围绕当前场次工作台继续实现。

## 2026-06-19 10:37：R2 暂缓

### 决策

Cloudflare R2 暂缓接入，截图云端存储不进入当前 MVP 必做范围。系统保留截图相关数据结构和后续扩展位置，但运行时不要求 `SCREENSHOTS` 绑定存在。

### 原因

R2 需要绑定银行卡，当前银行卡暂不支持。截图管理是辅助功能，不影响账号、主播、粉丝、场次、定榜、存票和锁牌等核心流程。

### 注意

后续如果恢复截图功能，需要重新启用 R2 或选择其它对象存储方案，并补充上传限制、删除策略和公开字段控制。

## 2026-06-19 09:58：Pages API 接入方式

### 决策

线上优先使用 Cloudflare Pages Functions 接入 `/api/*`，复用现有 Hono Worker 代码。独立 Worker 配置继续保留，作为本地调试和后续拆分 API 的备选。

### 原因

- Pages Functions 可以让前端和 API 共用 `jcc-web-mvp.pages.dev` 同一个域名。
- 同域名 cookie 更简单，适合当前登录方案。
- 不需要额外配置 Worker 路由或跨域 API 域名。
- 当前 Cloudflare Pages 后台已经提供 Functions 绑定入口，可直接绑定 D1/R2/环境变量。

### 注意

Cloudflare Pages 后台必须配置 `DB` 和 `ADMIN_SETUP_TOKEN`。`SCREENSHOTS` 因 R2 暂缓改为可选绑定。

## 2026-06-19 09:31：账号会话策略

### 决策

账号登录使用服务端会话表 `account_sessions`，前端只保存 HTTP-only cookie。MVP 先实现单账号单设备在线：同账号新登录会撤销旧会话。

生产环境首次初始化管理员前，需要配置 `ADMIN_SETUP_TOKEN`。

### 原因

- 粉丝不登录，登录主体只有管理员和主播，服务端会话足够清晰。
- 单账号单设备在线符合当前需求，能降低多设备同时操作造成的数据冲突。
- HTTP-only cookie 比把 token 放在前端状态或 localStorage 中更稳妥。
- 管理员初始化是高风险入口，线上环境必须有初始化口令。

### 注意

后续实现主播账号管理时，需要继续沿用 `account_sessions` 和 `audit_logs`，所有停用、重置、代操作都要保留日志。

## 2026-06-18：技术栈

### 决策

采用：

- TypeScript。
- React。
- Vite。
- Drizzle ORM。
- SQLite 本地起步。
- Cloudflare Workers / D1 / R2 优先适配。
- Docker 云服务器作为备选。

### 原因

- TypeScript 能降低复杂业务字段和票务计算出错概率。
- React 适合后台管理页面。
- SQLite/D1 有利于本地和 Cloudflare 路线衔接。
- Drizzle 对 SQL 表结构和迁移较清晰。
- 保留 Docker 备选，避免被 Cloudflare 平台限制卡死。

## 2026-06-18：UI 风格

### 决策

先做朴素后台风格，优先功能完整和操作清楚。

### 原因

本项目是直播辅助工具，不是营销展示页。高频操作、低错误率、可扫视比视觉复杂度更重要。

## 2026-06-18：公开策略

### 决策

游客只看公开模块和公开字段。公开确认采用模块级开关。

### 原因

降低误公开风险，同时避免逐条审核带来的操作负担。

## 2026-06-18：票务策略

### 决策

票务以流水为真值，不直接硬改历史余额。修改使用修正记录，删除使用作废记录。

### 原因

存票是项目最敏感的数据之一，必须可追溯。

## 2026-06-18：npm / pip 代理

### 决策

当前本机 npm / pip 均统一使用 Clash Verge 混合代理端口：

```text
http://127.0.0.1:7897
```

### 原因

旧配置中存在 `7890` 和 `10809`，与当前 Clash Verge 端口不一致，导致依赖下载失败。

### 注意

如果 Clash Verge 端口变化，需要同步调整 npm / pip 配置。

## 2026-06-18 18:22：提交节奏

### 决策

每完成一组明确任务后，应创建一次 Git 提交。

### 原因

项目后续会涉及数据库迁移、票务规则、权限、公开配置和部署适配。及时提交能保留清晰版本点，方便回滚和追踪问题来源。

### 执行原则

- 不做每行小改后的机械提交。
- 不提交半成品调试噪音。
- 提交前检查状态并运行必要验证。
- 用户明确要求暂不提交时，以用户要求为准。

## 2026-06-18 23:31：Cloudflare Pages 发布目录

### 决策

Cloudflare Pages 必须发布 Vite 构建产物 `dist`，不能继续发布仓库根目录。

### 原因

新项目根目录 `index.html` 引用 `/src/main.tsx`，这是开发入口，不是生产产物。生产环境应发布 `npm run build` 生成的 `dist`。

### 执行要求

Cloudflare Pages 设置：

```text
Build command: npm run build
Build output directory: dist
Root directory: /
```

仓库中保留 `pages_build_output_dir = "dist"`，用于 Wrangler Pages 适配。

## 2026-06-18 23:41：Pages 和 Worker 配置拆分

### 决策

`wrangler.toml` 仅用于 Cloudflare Pages。独立 Worker API 使用 `wrangler.worker.toml`。

### 原因

Pages 当前只需要构建和发布 Vite 的 `dist`。Worker 的 `main`、D1、R2 占位配置不应影响 Pages 自动部署。后续需要 Worker API 时，再用独立配置处理。

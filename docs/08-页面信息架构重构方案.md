# 08-页面信息架构重构方案

整理时间：2026-06-20 00:08

## 当前判断

当前项目已经具备账号、主播、粉丝、场次、票务流水和定榜基础能力，但前端页面开始偏向“一个模块一个管理页”的后台 CRUD 结构。

这种结构适合管理员维护数据，但不适合主播直播中高频操作。主播真实流程不是先去“场次页”，再跳到“定榜页”，再跳到“存票页”，再跳到“锁牌页”；她更需要围绕当前直播场次的一体化操作台。

因此下一阶段建议先暂停继续增加锁牌、结算等新功能，优先调整页面信息架构。后端和已有 API 不需要推倒重来，重点是重组前端体验和操作路径。

## 设计目标

- 主播端：直播中少跳页、少选择、少误操作，优先服务“当前场次”。
- 管理端：能快速代操作、查资料、查流水、查日志、处理异常。
- 游客端：只展示公开允许的内容，重点是公开存票榜。
- 数据层：继续沿用当前账号、主播、粉丝、场次、票务和定榜基础。
- UI 风格：仍保持朴素后台风格，但页面结构要更贴近真实业务流程。

## 核心原则

### 1. 当前场次优先

主播登录后不应先看到一组分散菜单，而是优先看到“今日工作台”和“当前场次”。

当前场次应成为定榜、名单、对局、锁牌、存票、结算、截图的共同上下文。只要进入当前场次，后续操作默认都挂到这个场次，避免反复选择场次。

### 2. 直播操作和资料维护分开

主播端优先是直播操作台，不是资料管理后台。

粉丝资料、主播账号、日志、历史流水等仍然需要，但它们应该更多出现在管理端，主播端只保留直播中必要的搜索、新建、备注和状态提醒。

### 3. 推荐不是裁决，确认才是结果

定榜推荐名单只能作为系统建议。实际现场可能有新粉不足、老粉顺延、粉丝协商补票、临时不玩、主播状态不好、双开等情况。

因此推荐名单之后必须有“名单确认”步骤，允许手动调整席位、待定、备注和回退。

### 4. 高风险操作需要预览和确认

票务结算、作废、拉黑、公开设置、账号停用、密码重置都应有明确提示。

直播中可以录得快，但真正影响长期余额和公开内容的操作，要给操作者一个确认机会。

### 5. 管理员代操作必须显眼

管理员代主播操作时，页面顶部需要一直显示“正在代操作：某主播”，并且所有关键操作写入日志。

这样可以避免管理员以为自己在看全局数据，实际却改了某个主播空间里的数据。

## 新页面结构

```text
登录/初始化
  -> 按身份进入

主播端
  -> 今日工作台
  -> 当前场次工作台
      -> 定榜
      -> 名单确认
      -> 对局/锁牌
      -> 存票
      -> 结算
      -> 截图/备注
  -> 我的资料/简单设置

管理端
  -> 管理首页
  -> 主播详情
      -> 当前场次代操作
      -> 粉丝资料
      -> 存票汇总
      -> 历史场次
      -> 日志
  -> 账号管理
  -> 资料管理
  -> 票务流水
  -> 操作日志
  -> 系统设置

游客端
  -> 公开首页
  -> 公开存票榜
  -> 公开截图/内容页（后续）
```

## 主播端设计

### 今日工作台

定位：主播打开系统后的第一屏。

核心内容：

- 当前登录主播。
- 今日是否有进行中场次。
- 没有场次时，显示创建下午场、晚上场、自定义场次。
- 有场次时，显示进入当前场次。
- 待处理提醒：未确认定榜、未完成对局、待结算、异常票务。
- 快捷搜索粉丝。

首屏不建议放大量历史数据。主播直播时最重要的是“现在要做什么”。

### 当前场次工作台

定位：直播期间的主操作页面。

建议结构：

```text
场次头部
  主播 / 场次标题 / 状态 / 模式 / 开始时间 / 待结算提示

阶段导航
  定榜 | 名单确认 | 对局/锁牌 | 存票 | 结算 | 截图/备注

主操作区
  根据当前阶段显示主要表单、列表和推荐

辅助区
  当前名单、票务提醒、粉丝状态提醒、最近操作
```

移动端可以把辅助区折叠到主内容下面，优先保证按钮和输入框好点、好看清。

### 定榜区

目标：录入一次定榜并快速看出谁应上车。

内容：

- 定榜类型：定榜七候选、定榜五顺延、手动。
- 榜上顺序。
- 粉丝。
- 礼物钻。
- 取票。
- 人工调整。
- 总票数。
- 状态：推荐上车、待定、禁赛、手动调整。
- 备注。

交互建议：

- 总票数实时显示：礼物钻 + 取票 + 调整。
- 推荐名单固定显示在录入区旁边或下方。
- 拉黑粉丝醒目标记，不占推荐席位。
- 保存榜单条目后自动重算推荐，但不自动创建对局。
- 点击“进入名单确认”后再决定最终席位。

### 名单确认区

目标：把系统推荐转成真实对局名单。

内容：

- 8 个席位：主播 + 粉丝位。
- 每个席位显示粉丝名、状态、新粉/老粉/管理/违规/拉黑提醒、备注游戏名。
- 待定池：没有上车但可能下一把或补位的人。
- 操作：上车、待定、取消、补位、备注。
- 确认后创建对局。

重点：

- 支持正常“定榜七候选，有新粉就后面待定”的风格。
- 支持“定榜五，没新粉就顺延老粉”的风格。
- 不把推荐名单当作最终结果，最终结果由主播或管理员确认。

### 对局/锁牌区

目标：一局游戏中快速记录锁牌变化。

内容：

- 8 个席位卡片或紧凑表格。
- 玩家状态：存活、淘汰、退赛、备注。
- 锁定五费。
- 剩余可用五费。
- 淘汰排名。
- 本局游戏名备注。

交互建议：

- 点击玩家后选择锁牌。
- 已被存活玩家锁的牌，对其他存活玩家置灰或提示占用。
- 淘汰后保留记录，但释放占用。
- 完成对局后提醒是否把本局新粉转为老粉。

### 存票区

目标：直播中快速处理存票、取票、现刷、修正。

内容：

- 粉丝搜索。
- 当前长期余额。
- 本场相关流水。
- 本次定榜已使用票。
- 新增记录：存票、取票、现刷、修正。
- 作废入口。
- 未使用取票回退提醒。

交互建议：

- 直播中允许快速记账，但长期余额变化必须可追溯。
- 修改取票金额建议通过新增修正或作废重录，保留日志。
- 对“取票未上车”提供标记，结算时回退。

### 结算区

目标：直播结束后清楚地确认本场票务。

内容：

- 本场所有存票、取票、现刷、修正汇总。
- 未使用取票列表。
- 需要回退的金额。
- 结算后粉丝余额变化预览。
- 确认结算按钮。

重点：

- 结算必须二次确认。
- 结算后状态变更为已结算。
- 结算结果写日志。

## 管理端设计

### 管理首页

定位：管理员的全局看板。

内容：

- 主播列表。
- 进行中场次。
- 待结算场次。
- 最近异常：负余额尝试、作废、拉黑、公开设置变更、登录异常。
- 快速进入某主播详情。

### 主播详情

定位：管理员管理某个主播的主入口。

内容：

- 主播资料和账号状态。
- 当前场次入口。
- 粉丝资料。
- 存票汇总。
- 历史场次。
- 最近操作日志。
- 管理员代操作入口。

进入代操作后，顶部全局提示：

```text
正在代操作：主播名称
```

### 资料管理

定位：低频维护。

内容：

- 粉丝资料。
- 赛季。
- 五费卡。
- 规则模板。
- 礼物字典（后续）。

资料页不应成为主播直播时的主要入口。

### 票务流水

定位：查账和纠错。

内容：

- 按主播、粉丝、场次、类型、时间筛选。
- 查看流水详情。
- 作废。
- 查看关联日志。

### 操作日志

定位：追溯问题。

内容：

- 操作者。
- 目标主播。
- 动作。
- 对象类型。
- 时间。
- 备注。
- 关键前后变化。

## 游客端设计

### 公开首页

内容：

- 主播公开名称。
- 可公开模块入口。
- 公开存票榜入口。
- 后续可扩展公开截图。

### 公开存票榜

内容：

- 排名。
- 公开粉丝名。
- 存票余额。
- 分页。
- 按公开字段筛选。

限制：

- 未公开抖音名时，游客不能按抖音名搜索。
- 不展示微信名、真实备注、违规备注、内部状态。
- 拉黑、违规等敏感信息默认不公开。

## 当前页面问题对应改法

| 当前问题 | 影响 | 调整方向 |
| --- | --- | --- |
| 导航按模块分散 | 主播直播中来回跳页 | 改为“当前场次工作台”统一承载 |
| 场次、定榜、存票互相割裂 | 容易选错场次或漏记 | 当前场次作为全局上下文 |
| 定榜页只有推荐结果 | 无法表达现场调整 | 增加名单确认区 |
| 管理端和主播端入口混在一起 | 不同身份操作重点不清楚 | 按角色重组导航 |
| CRUD 表单占比过高 | 管理能用，直播不顺手 | 高频操作做成紧凑工作流 |
| `src/App.tsx` 过大 | 后续维护困难 | 前端重构时拆分页面和组件 |

## 现有能力复用

可以继续复用：

- 账号登录和会话。
- 管理员创建、编辑、停用、重置主播账号。
- 粉丝资料和状态字段。
- 直播场次基础。
- 票务流水和余额重算。
- 定榜快照和条目。
- 总票数计算和推荐基础。
- 审计日志表和部分日志写入。
- Cloudflare Pages Functions + D1 部署方式。

需要调整或补充：

- 前端导航和页面组织。
- 当前场次上下文。
- 名单确认数据模型和 API。
- 对局/席位/锁牌数据模型和 API。
- 结算预览和确认。
- 游客公开存票榜接口。
- 操作日志筛选页。

## 实施顺序建议

### 第一步：前端信息架构重组

- 新增角色化导航。
- 新增主播“今日工作台”。
- 新增“当前场次工作台”壳子。
- 将现有定榜、存票、场次能力先嵌入当前场次上下文。
- 管理端保留现有资料管理入口。

验收：

- 主播登录后能从今日工作台进入当前场次。
- 当前场次内能看到定榜、存票、对局/锁牌、结算入口。
- 管理员能进入主播详情并代操作。

### 第二步：名单确认

- 定榜推荐后进入名单确认。
- 支持 8 席位调整、待定、备注。
- 确认后创建对局草稿。

验收：

- 能从一次定榜生成可调整名单。
- 能手动调整上车/待定。
- 拉黑粉丝不能进入最终名单。

### 第三步：对局和锁牌

- 创建对局。
- 8 人席位。
- 锁牌互斥。
- 淘汰释放占用。
- 完成对局后新粉转老粉提醒。

验收：

- 一局规则水友赛可完整记录锁牌。
- 剩余五费显示正确。

### 第四步：结算和公开榜

- 本场结算预览。
- 未使用取票回退。
- 确认结算。
- 游客公开存票榜。

验收：

- 结算前后余额可解释。
- 游客只能看到公开字段。

## 暂停事项

在信息架构重组完成前，建议暂停：

- 继续实现完整锁牌页面。
- 继续增加截图管理。
- 增加复杂统计大屏。
- 增加礼物字典和小计算器。
- 大规模美化 UI。

这些功能不是不要做，而是应等主流程页面定型后再接入，避免二次返工。

## 结论

当前不是后端基础失败，而是前端页面组织到了需要校准的时候。下一阶段应把系统从“功能列表”调整为“直播操作台 + 管理后台 + 游客公开页”。

这能让主播端更顺手，也能让管理端保持强管理能力。后续继续做锁牌、结算、公开榜时，应全部围绕这个新架构推进。

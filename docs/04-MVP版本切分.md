# 04-MVP版本切分

整理时间：2026-06-18

## 切分原则

MVP 不是把所有想法做完，而是优先跑通真实直播核心链路：

```text
创建场次
  -> 定榜
  -> 生成/确认名单
  -> 对局锁牌
  -> 存票/取票/现刷
  -> 结算
  -> 游客看公开存票榜
```

第一版必须避免后期大改的数据结构，所以账号、主播、粉丝、场次、票务流水、日志和截图存储要从一开始设计对。

## MVP 必须做

### 1. 账号和权限

- 管理员登录。
- 主播登录。
- 管理员创建主播账号。
- 粉丝不登录、不注册。
- 单账号单设备在线策略。

### 2. 主播和粉丝资料

- 主播资料。
- 粉丝资料：昵称、抖音名、微信名、游戏名、备注。
- 粉丝状态：新粉、老粉、管理、违规、拉黑。
- 拉黑禁赛。
- 新粉可手动或对局后转老粉。

### 3. 直播场次

- 创建场次。
- 下午场/晚上场/自定义。
- 进行中、待结算、已结算状态。
- 规则水友赛和无规则模式。

### 4. 定榜和名单

- 录入定榜条目。
- 支持礼物钻、取票、人工调整。
- 总竞争票默认计算。
- 支持定榜七候选和定榜五顺延两种风格。
- 生成推荐名单。
- 主播确认名单并可手动调整。

### 5. 对局锁牌

- 规则水友赛 8 个席位。
- 记录本局游戏名/备注。
- 锁牌互斥。
- 淘汰保留记录并释放占用。
- 剩余五费显示。
- 保存对局历史。

### 6. 存票和结算

- 存票、取票、现刷、修正、回退。
- 本场预览余额。
- 直播结束结算确认。
- 作废票务流水。
- 所有票务变化记录日志。

### 7. 公开存票榜

- 模块级公开开关。
- 公开粉丝余额排名。
- 分页和筛选。
- 只能按公开字段筛选。

### 8. 截图基础

- 上传到服务器或对象存储。
- 关联场次、对局、粉丝。
- 类型和备注。
- 公开状态。
- 删除错误图片。
- 单张 5MB 推荐限制。

### 9. 操作日志

- 登录外的关键操作必须记录。
- 票务、结算、作废、拉黑、公开设置、截图删除、管理员代操作。

## MVP 可以做

- 礼物字典和票数小计算器。
- 管理端异常提醒。
- 数据导出。
- 公开截图页。
- 规则模板管理界面。
- 更精细的图片压缩。

## MVP 暂不做

- 自动读取抖音榜单。
- 自动识别截图文字。
- 粉丝登录/注册。
- 支付、提现、分账。
- 多设备同时编辑。
- 一键清空全部存票。
- 完整运营统计大屏。
- OBS 直播展示页。

## 第一版验收标准

第一版可认为完成，当它能支持：

- 管理员创建主播账号。
- 主播创建一场直播。
- 主播录入一次定榜并生成名单。
- 主播创建一局规则水友赛并完成锁牌记录。
- 主播记录存票、取票、现刷和修正。
- 主播结算本场并更新余额。
- 管理员能代操作并看到日志。
- 游客能看到公开存票榜。
- 截图能上传、关联、公开或删除。

## 风险点

- 票务规则容易因现场例外变复杂，所以必须保留人工调整和备注。
- 公开字段如果处理不好，容易泄露抖音名、备注或违规记录。
- 图片存储会影响免费部署额度，所以要限制体积。
- 锁牌和淘汰规则需要测试，避免释放占用逻辑出错。
- 结算确认必须清晰，否则会误改长期余额。

# 10-页面交互设计方案B

整理时间：2026-06-20 18:16

## 文档目标

本文用于梳理当前项目下一阶段的页面交互设计。

本次暂定采用 `方案B：清晰重排版`，核心方向是：

```text
当前场次优先
  -> 实时榜单为主
  -> 快速编辑为辅
  -> 定榜冻结形成结果
  -> 名单确认承接下一步
  -> 场次结束后统一结算
```

本文不直接写实现代码，而是先确认页面目标、操作路径、状态流转、异常处理和需要用户确认的问题。

## 当前判断

当前项目已经具备账号、主播、粉丝、直播场次、票务流水、本场实时榜单、定榜倒计时和冻结能力，但页面和流程仍有关键心智偏差。

### 1. 场次状态不是清晰操作链

当前已有场次状态：

```text
preparing / live / pending_settlement / settled / cancelled
```

但页面上更像是“创建场次后记录一个状态”，还没有形成清楚的操作链：

```text
创建场次
  -> 开始直播
  -> 结束直播
  -> 确认结算
  -> 已结算
```

真实使用时，主播更需要明确按钮，而不是只看到状态字段。

### 2. 本场榜单里的存票/取票不应立即正式入账

当前本场榜单保存时，如果 `取票` 或 `存票` 有变化，会按差额直接写入 `ticket_ledgers`，并刷新粉丝长期余额。

这与已确认的业务心智不一致。

更合理的理解是：

```text
直播中录入的礼物钻、取票、存票、调整
  = 本场榜单草稿 / 本场结算草稿

直播结束并确认结算后
  = 正式写入长期票务流水并更新余额
```

因此，本场榜单保存时不应直接改变长期余额。它应只影响本场总票和结算预览。

### 3. 定榜冻结后下一步不够突出

当前定榜冻结后，页面仍然容易停留在“重新冻结 / 开始三分钟”这一类操作上。

更符合心智的主路径应该是：

```text
冻结成功
  -> 查看冻结结果
  -> 进入名单确认
  -> 创建对局
```

重新冻结是补救动作，不应成为冻结后的主操作。

### 4. 页面仍有后台 CRUD 痕迹

项目已经开始从模块页面转向当前场次工作台，但旧的 `场次管理`、`定榜管理`、`存票管理` 入口仍会让用户误以为每个功能都是孤立页面。

下一阶段应把主播端高频操作收拢到当前场次工作台，把低频维护留给管理页或资料页。

## 页面总目标

主播端页面目标：

```text
帮助主播在一场直播中连续完成榜单维护、定榜冻结、名单确认、对局记录和场次结算。
```

管理端页面目标：

```text
帮助管理员选择主播、代操作当前场次、维护资料、排查票务和查看日志。
```

游客端页面目标：

```text
只展示已允许公开的存票榜和公开内容，不暴露内部备注、违规信息和未公开字段。
```

## 总体页面结构

```text
登录 / 初始化
  -> 按角色进入

主播端
  -> 今日工作台
  -> 当前场次工作台
      -> 榜单/定榜
      -> 名单确认
      -> 对局/锁牌
      -> 结算
      -> 存票流水
      -> 截图/备注

管理端
  -> 管理首页
  -> 选择主播
  -> 代操作当前场次
  -> 资料管理
  -> 票务流水
  -> 操作日志

游客端
  -> 公开首页
  -> 公开存票榜
```

## 场次生命周期设计

### 状态定义

| 状态 | 用户看到的文案 | 含义 | 主要操作 |
| --- | --- | --- | --- |
| `preparing` | 准备中 | 场次已创建，但直播尚未正式开始 | 开始直播、取消场次 |
| `live` | 进行中 | 正在直播，可以录榜、定榜、对局 | 结束直播、继续操作 |
| `pending_settlement` | 待结算 | 直播已结束，等待确认票务 | 查看结算预览、确认结算、返回修改 |
| `settled` | 已结算 | 长期余额已正式入账 | 查看结果、结算后修正 |
| `cancelled` | 已取消 | 场次不再参与正常流程 | 查看或隐藏 |

### 推荐操作链

```text
创建场次
  -> 状态为准备中
  -> 点击开始直播
  -> 状态为进行中，记录 started_at
  -> 直播中维护榜单、定榜、对局
  -> 点击结束直播
  -> 状态为待结算，记录 ended_at
  -> 查看结算预览
  -> 确认结算
  -> 状态为已结算，记录 settled_at 和确认人
```

### 页面交互

准备中：

- 主按钮：`开始直播`
- 次按钮：`编辑场次信息`
- 危险按钮：`取消场次`

进行中：

- 主按钮：`结束直播`
- 次按钮：`榜单/定榜`、`名单确认`、`对局/锁牌`
- 页面允许编辑本场榜单和定榜。

待结算：

- 主按钮：`确认结算`
- 次按钮：`查看结算预览`、`返回修改本场榜单`
- 页面应明确提示：确认后才会写入长期余额。

已结算：

- 主按钮：`查看结算结果`
- 次按钮：`结算后修正`
- 默认不允许继续随意修改本场榜单。

## 当前场次工作台

当前场次工作台是主播端主页面。

### 顶部总控条

顶部总控条固定回答 4 个问题：

- 当前是哪位主播。
- 当前是哪场直播。
- 当前处于哪个阶段。
- 下一步最重要的操作是什么。

建议显示：

- 主播名称。
- 场次标题。
- 场次类型：下午场 / 晚上场 / 自定义。
- 场次状态。
- 开始时间、结束时间。
- 当前榜单人数。
- 当前定榜状态。
- 待处理提醒。

根据状态显示主按钮：

| 场次状态 | 主按钮 |
| --- | --- |
| 准备中 | 开始直播 |
| 进行中 | 结束直播 |
| 待结算 | 确认结算 |
| 已结算 | 查看结算结果 |

管理员代操作时，顶部必须持续显示：

```text
正在代操作：主播名称
```

### 阶段导航

建议保留阶段导航，但不要让它像普通模块菜单。

推荐顺序：

```text
榜单/定榜 | 名单确认 | 对局/锁牌 | 结算 | 存票流水 | 截图/备注
```

调整原因：

- 榜单/定榜是直播中最先发生的高频动作。
- 名单确认应紧接冻结结果。
- 对局/锁牌承接名单。
- 结算是场次结束后的主动作。
- 存票流水更多是查看和追溯，不应打断主链路。

## 榜单/定榜页面

本页面采用方案B布局。

```text
顶部：当前场次总控条

中间左侧：实时榜单主区域

中间右侧：操作面板
  - 快速编辑
  - 定榜控制
  - 冻结结果摘要

底部：定榜记录 / 历史冻结记录
```

### 实时榜单主区域

实时榜单是主视觉，不再让快速编辑表单占主位。

建议列：

| 列 | 说明 |
| --- | --- |
| 排名 | 按本场总票排序 |
| 粉丝 + 本场总票 | 主视觉，方便截图和扫视 |
| 标签 | 新粉、老粉、管理、违规、拉黑 |
| 票数明细 | 礼物钻 + 取票 - 存票 + 调整 |
| 余额预览 | 当前长期余额 -> 预计结算后余额 |
| 状态 | 正常、新粉、待定、有事不来、禁赛 |
| 备注 | 本场备注 |
| 操作 | 编辑、标记有事、待定、恢复 |

### 分区规则

推荐分区：

```text
正常竞争榜
本场新粉
待定
有事不来
禁赛/拉黑
```

注意：

- 正常竞争榜是主区。
- 本场新粉单独显示，便于主播判断新粉数量。
- 有事不来不删除，只从竞争榜移到特殊分区。
- 禁赛/拉黑不能进入推荐名单。

### 快速编辑

快速编辑是右侧工具面板。

操作路径：

```text
搜索/选择粉丝
  -> 编辑礼物钻、取票、存票、调整
  -> 选择状态
  -> 填同票顺序和备注
  -> 保存本场榜单
  -> 实时榜单重排
```

字段：

- 粉丝。
- 礼物钻。
- 取票。
- 存票。
- 调整。
- 状态。
- 同票顺序。
- 备注。

保存后应只更新：

- `live_session_board_entries`。
- 本场总票。
- 结算预览。
- 操作日志。

保存后不应立即正式写入长期票务流水。

### 票数和余额口径

本场总票：

```text
礼物钻 + 取票 - 存票 + 调整
```

结算后余额预览：

```text
当前长期余额 - 取票 + 存票 + 回退 + 余额调整
```

页面应明确区分：

- `当前长期余额`：上次已结算后的真实余额。
- `本场结算预览`：本场如果现在结算，会变成多少。
- `本场总票`：参与本场榜单竞争的票。

### 定榜控制

右侧操作面板中包含定榜控制。

未开始定榜时：

- 定榜编号。
- 定榜规则：定榜七 / 定榜五。
- 倒计时，默认 3 分钟。
- 定榜名称。
- 主按钮：`开始定榜倒计时`。

倒计时中：

- 大号倒计时。
- 提示：倒计时中仍可修改榜单。
- 主按钮：`手动冻结`。
- 次按钮：`取消倒计时`。

已冻结：

- 显示冻结时间。
- 显示冻结人数。
- 显示推荐名单摘要。
- 主按钮：`进入名单确认`。
- 次按钮：`重新冻结当前榜单`。

### 定榜记录

底部定榜记录用于追溯，不作为主操作区。

每条记录显示：

- 第几次定榜。
- 定榜名称。
- 规则。
- 状态。
- 冻结时间。
- 操作者。
- 是否已进入名单确认。
- 是否已创建对局。

操作：

- 查看详情。
- 进入名单确认。
- 重新冻结。
- 作废，后续可做。

## 名单确认页面

名单确认是定榜冻结后的必经步骤。

### 目标

把系统推荐名单变成真实对局名单。

定榜推荐不是最终裁决，主播/管理员确认后才进入对局。

### 页面结构

```text
顶部：来源定榜摘要
中间：8 个席位
右侧：待定池 / 新粉 / 有事不来 / 禁赛提醒
底部：确认创建对局
```

### 8 个席位

默认结构：

```text
主播位
老粉位 1
老粉位 2
老粉位 3
老粉位 4
老粉位 5
新粉位 1
新粉位 2
```

但不能写死。页面应允许：

- 没有新粉时，老粉顺延。
- 老粉不足时，等待或补位。
- 主播状态不好时，只带老粉。
- 无规则模式不强制 8 人。
- 小号/多开通过备注记录。

### 操作

- 接受推荐名单。
- 替换席位。
- 把粉丝设为待定。
- 从待定池补位。
- 标记有事不来。
- 记录本局游戏名。
- 记录备注。
- 确认名单并创建对局。

### 风险控制

- 拉黑粉丝不能进入席位。
- 有事不来的粉丝默认不能进入席位，除非恢复状态。
- 确认名单后写操作日志。
- 如果来源定榜尚未冻结，不允许确认名单。

## 对局/锁牌页面

### 目标

一局游戏中快速记录 8 人席位、锁牌、淘汰和剩余五费。

### 页面结构

```text
顶部：当前对局信息
中间：8 个玩家席位
右侧：五费卡池 / 剩余可用
底部：操作记录和完成对局
```

### 操作

- 设置本局模式：规则水友赛 / 无规则模式。
- 设置本局游戏名或备注。
- 给玩家锁五费。
- 淘汰玩家。
- 记录最终排名。
- 完成对局。
- 完成后提醒是否把本局新粉转为老粉。

### 心智规则

- 存活玩家之间锁牌互斥。
- 淘汰玩家保留锁牌记录，但释放占用。
- 无规则模式不进入锁牌流程，但可以保存参与人和备注。

## 结算页面

结算页面是当前最大缺口之一。

### 目标

直播结束后，把本场所有票务草稿转成正式长期流水。

### 入口条件

场次状态为：

```text
pending_settlement
```

也就是已经点击 `结束直播`。

进行中场次可以预览结算，但不建议允许确认结算。

### 页面结构

```text
顶部：场次结算状态
中间：粉丝级结算明细
右侧：异常提醒
底部：确认结算
```

### 粉丝级结算明细

每个粉丝显示：

- 当前长期余额。
- 本场礼物钻。
- 本场取票。
- 本场存票。
- 未使用取票回退。
- 余额调整。
- 预计结算后余额。
- 是否参与对局。
- 备注。

### 正式入账规则

确认结算时生成正式流水：

| 本场字段 | 正式流水 | 是否影响余额 | 说明 |
| --- | --- | --- | --- |
| 取票 | `withdraw` | 是 | 从长期余额扣除 |
| 存票 | `deposit` | 是 | 加入长期余额 |
| 礼物钻 | 不默认入账 | 否 | 只影响本场总票 |
| 未使用取票回退 | 可不扣或生成回退记录 | 是 | 需要明确记录 |
| 余额调整 | `adjustment` | 是 | 人工修正 |

确认结算后：

- 更新粉丝缓存余额。
- 场次状态变为已结算。
- 写入结算确认人。
- 写操作日志。

### 待确认规则

需要确认：

- 如果取票后没有上车，默认是否自动回退？
- 如果取票后上过一局，是否一定不回退？
- 如果粉丝只待定但没有玩，是否回退？
- 如果直播中途有事离开，按本场、按定榜、还是按对局判断是否回退？

## 存票流水页面

存票流水页面不应作为直播主操作页。

它更适合作为：

- 查账。
- 补录历史存票。
- 作废错误流水。
- 做结算后修正。

### 操作

- 查粉丝流水。
- 按场次过滤。
- 新增历史补录。
- 新增余额修正。
- 作废流水。
- 查看关联场次或定榜。

### 心智边界

直播中的本场存票/取票优先在榜单页录入草稿。

历史补录、结算后修正、错误作废才进入存票流水页。

## 今日工作台

今日工作台是主播登录后的第一屏。

### 目标

让主播快速回到当前直播。

### 内容

- 当前进行中场次。
- 待结算场次。
- 快速创建下午场/晚上场/自定义场次。
- 最近一次未完成动作。

### 操作

- 创建场次，默认状态建议为 `preparing`。
- 进入当前场次。
- 进入待结算场次。

注意：

当前如果创建场次后直接设为 `live`，会弱化 `开始直播` 操作。建议调整为创建后先 `preparing`，再由用户点击 `开始直播`。

## 管理端

管理端重点不是直播操作本身，而是选择主播、代操作、维护和排查。

### 管理首页

显示：

- 主播列表。
- 进行中场次。
- 待结算场次。
- 最近异常。
- 快速进入代操作。

### 代操作

管理员进入主播当前场次后，应持续显示：

```text
正在代操作：主播名称
```

所有关键操作写日志：

- 场次开始/结束。
- 榜单修改。
- 定榜冻结。
- 名单确认。
- 对局创建。
- 结算确认。
- 票务作废。
- 拉黑/解除拉黑。

## 游客公开页

游客端只做公开内容。

### 公开存票榜

显示：

- 排名。
- 公开粉丝名。
- 公开余额。
- 分页。
- 公开允许的筛选字段。

不显示：

- 内部昵称，除非设置为公开名。
- 微信名。
- 未公开抖音名。
- 内部备注。
- 违规备注。
- 拉黑原因。
- 票务流水明细，除非后续单独公开。

### 公开开关

建议公开采用模块级开关：

- 是否开启公开存票榜。
- 是否只展示标记公开的粉丝。
- 是否允许显示精确余额。
- 是否允许搜索公开名。

## 发现的其他流程问题

### P0：本场榜单保存立即写长期流水

问题：

直播中编辑存票/取票会直接影响长期余额。

建议：

改为本场草稿，结算确认后正式写流水。

### P0：缺少明确结算确认页

问题：

已有 `settled_at` 和 `settlement_confirmed_by` 字段，但页面流程尚未完整体现结算预览、异常提醒和确认。

建议：

先实现结算预览，再允许确认结算。

### P1：创建场次直接进入进行中

问题：

快速创建当前会让用户感觉创建就是开播，缺少准备态。

建议：

创建后默认为准备中，用户点击开始直播后进入进行中。

如果主播确实需要一键开播，可按钮文案明确为 `创建并开始直播`。

### P1：已结算后是否允许修改不清楚

问题：

如果已结算后继续修改本场榜单，会破坏结算结果解释。

建议：

已结算场次默认只读。修改应走结算后修正或重新打开结算，但需要权限和日志。

### P1：冻结快照缺少独立存票字段

问题：

当前冻结快照能保证总票正确，但存票被折入调整，不利于复盘。

建议：

后续给 `ranking_entries` 增加 `ticket_deposit` 字段。

### P1：名单确认数据模型还未落地

问题：

冻结后到创建对局之间缺少正式承接对象。

建议：

至少需要表达：

- 来源定榜。
- 8 个席位。
- 待定池。
- 手动调整记录。
- 确认人和确认时间。

可以直接创建 `game_matches` 和 `match_seats` 草稿，也可以新增名单确认表。需要进一步评估。

### P1：本场调整的含义需要拆分

问题：

当前 `调整` 同时可能表示竞争票调整，也可能表示余额调整。

建议：

页面上至少区分：

- `竞争票调整`：只影响本场总票。
- `余额调整`：影响结算后长期余额。

MVP 可以先只保留竞争票调整，余额调整放到结算页或存票流水页。

### P1：存票页面和榜单页面职责容易重叠

问题：

两个页面都能处理存票/取票时，用户不知道该在哪录。

建议：

- 直播中的本场取票/存票：榜单页。
- 历史补录和结算后修正：存票流水页。
- 结算确认：结算页。

### P2：状态标签来源不统一

问题：

粉丝资料有新粉/老粉/管理/违规/拉黑，本场榜单也有正常/新粉/待定/有事/禁赛。

建议：

页面明确：

- 粉丝资料状态是长期身份。
- 本场榜单状态是本场参与状态。

例如：长期老粉也可以本场有事不来。

### P2：截图页暂缓但导航存在

问题：

R2 暂缓，截图/备注入口存在会让用户期待可上传。

建议：

第一版可改为 `备注` 或弱化截图入口。等存储方案可用后再恢复截图上传。

## 需要用户确认的问题

### 1. 场次创建后默认状态

建议：

```text
创建后默认准备中
```

需要确认是否保留一个快捷按钮：

```text
创建并开始直播
```

### 2. 取票何时正式扣余额

建议：

```text
结算确认时正式扣除
```

直播中只做预览。

### 3. 存票何时正式加余额

建议：

```text
结算确认时正式加入长期余额
```

直播中只做预览。

### 4. 取票未上车是否默认回退

需要确认默认规则。

候选：

- 未上车默认回退。
- 必须手动标记回退。
- 只要参与过对局就不回退，没参与才回退。

### 5. 调整字段是否拆分

建议拆成：

- 竞争票调整。
- 余额调整。

MVP 可先只在榜单页保留竞争票调整。

### 6. 已结算场次是否允许重开

需要确认是否允许：

```text
已结算 -> 重新打开结算
```

如果允许，应限制为管理员或二次确认，并写日志。

### 7. 名单确认是否需要独立保存历史

需要确认：

- 名单确认只是创建对局前的临时步骤。
- 还是需要单独保存每次名单确认记录。

如果经常需要复盘“为什么这个人没上车”，建议保存名单确认记录。

### 8. 管理员代操作是否可以确认结算

当前需求倾向可以。

需要确认是否所有主播都允许管理员代结算，还是后续需要权限开关。

## 当前结论

方案B可以继续采用，但必须同步修正流程心智：

```text
榜单页负责本场实时记录
定榜负责冻结某个时间点
名单确认负责把推荐变成席位
结算页负责正式入账
存票流水页负责查账和修正
```

其中最优先修正的是：

```text
直播中保存榜单，不应立即正式写长期票务流水。
```

这是当前最影响业务信任和后续页面解释的流程问题。

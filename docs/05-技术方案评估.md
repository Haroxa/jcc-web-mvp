# 05-技术方案评估

整理时间：2026-06-18

## 技术目标

项目需要同时满足：

- 本地可运行。
- 后续可部署在线。
- 支持服务器数据存储。
- 支持截图上传。
- 支持管理员和主播登录。
- 支持游客公开页。
- 未来能扩展多主播。

不建议做纯静态本地存储工具，因为它无法可靠支持多设备、截图、日志和在线公开页。

## 推荐架构方向

建议采用前后端一体的 Web 应用：

```text
浏览器
  -> 前端页面
  -> API 服务
  -> 数据库
  -> 图片存储
```

本地开发时：

```text
本地前端 + 本地 API + 本地数据库 + 本地文件存储
```

线上部署时：

```text
Cloudflare Pages/Workers + D1 + R2
或
Docker Web 服务 + 数据库 + 文件/对象存储
```

## 方案 A：Cloudflare 免费/低成本路线

组成：

- 前端：Cloudflare Pages。
- API：Cloudflare Workers。
- 数据库：Cloudflare D1。
- 截图：Cloudflare R2。

优点：

- 免费或低成本起步。
- 不需要自己维护服务器。
- 适合单主播 MVP。
- 前端公开页部署方便。

注意：

- Workers 有请求和执行限制，不适合高频实时轮询。
- D1 适合轻量结构化数据，不适合存图片本体。
- R2 适合截图，但要限制图片大小和数量。
- 本地开发需要准备一套兼容方案，避免被平台绑定死。

适合场景：

- 单主播。
- 低访问量。
- 管理员和主播少量操作。
- 游客主要查看公开存票榜。

## 方案 B：Docker 云服务器路线

组成：

- 一个 Web 服务。
- 数据库：PostgreSQL 或 SQLite 起步。
- 图片：服务器磁盘或对象存储。
- 反向代理：Nginx/Caddy 可选。

优点：

- 控制权更高。
- 部署和调试直观。
- 后续功能不容易受平台限制。
- 适合你买便宜服务器后长期使用。

注意：

- 需要维护服务器。
- 需要处理备份。
- 需要处理 HTTPS、域名、磁盘空间。
- 如果图片多，磁盘管理要谨慎。

适合场景：

- Cloudflare 限制不够用。
- 需要更复杂后端能力。
- 需要更自由的数据处理。

## 方案 C：只做本地应用

不推荐作为正式方向。

原因：

- 无法支持游客公开页。
- 多设备困难。
- 截图管理不可靠。
- 管理员无法远程辅助。
- 数据丢失风险高。

可以作为开发和应急运行能力保留，但不作为最终形态。

## 推荐选择

第一阶段推荐：

```text
架构按 Web 后端 + 数据库 + 文件存储设计
实现上优先保证本地可运行
部署上优先验证 Cloudflare 免费路线
保留 Docker 云服务器备选
```

这样不会过早绑定平台，也不会退回旧项目那种本地存储模式。

## 技术设计建议

### 数据库

第一版建议使用关系型结构。

原因：

- 票务流水需要强追溯。
- 场次、定榜、对局、粉丝关系清晰。
- 后续统计和筛选方便。

可选：

- Cloudflare D1：线上免费路线。
- SQLite：本地开发和轻量部署。
- PostgreSQL：Docker 云服务器长期路线。

### 图片存储

建议抽象为统一接口：

- 本地运行：保存到本地目录。
- Cloudflare：保存到 R2。
- Docker：保存到服务器磁盘或对象存储。

MVP 限制：

- 单张 5MB。
- 支持常见图片格式。
- 上传失败给明确提示。

### 登录和会话

MVP：

- 管理员和主播登录。
- 单账号单设备在线。
- 新设备登录后旧设备下线。

后续：

- 更细权限。
- 操作确认。
- 二次验证。

### 日志

日志是核心功能，不是后续补丁。

建议从第一版就记录：

- 操作者。
- 操作对象。
- 操作前后变化。
- 备注。
- 时间。

### 测试重点

优先测试：

- 存票余额计算。
- 取票/回退/结算。
- 作废流水。
- 锁牌占用释放。
- 公开字段筛选。
- 拉黑禁赛。

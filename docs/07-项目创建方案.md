# 07-项目创建方案

整理时间：2026-06-18

## 目标

本方案用于指导后续正式创建新项目。当前阶段仍不创建代码项目，只确定推荐技术栈、目录结构、运行方式和部署路线。

## 已确认技术路线

已确认采用：

- TypeScript + React。
- Drizzle ORM。
- SQLite 起步，兼容 Cloudflare D1。
- 优先 Cloudflare 适配，同时保证本地可运行。
- UI 先做朴素后台风格，优先功能完整。

## 推荐技术路线

推荐使用一个现代 Web 全栈项目，优先满足：

- 本地可运行。
- 后续可以部署到 Cloudflare。
- 也能切换到 Docker 云服务器。
- 数据结构清晰。
- 前端页面适配手机、平板、电脑。

## 推荐方案

### 首选：TypeScript + React + 后端 API + SQL 数据库

建议组合：

- 前端：React + TypeScript。
- 样式：Tailwind CSS 或轻量组件库。
- 后端：Node.js API 或 Cloudflare Workers API。
- 数据库：SQLite/D1 起步。
- ORM/查询层：Drizzle ORM。
- 图片存储：本地文件适配器 + R2 适配器。

理由：

- TypeScript 能减少业务字段和票务计算出错。
- React 适合做复杂管理页面。
- Drizzle 对 SQLite/D1/PostgreSQL 的路线比较友好。
- 本地 SQLite 和 Cloudflare D1 概念接近，便于 MVP 起步。
- 存储抽象后，本地文件和 R2 可切换。

## 备选方案

### 方案 B：Next.js 全栈

优点：

- 前后端一体。
- 页面、API、部署生态成熟。
- 本地开发舒服。

注意：

- 部署到 Cloudflare 需要额外适配。
- 如果未来走普通服务器或 Vercel 类平台更自然。

### 方案 C：纯 Node.js + React 分离

优点：

- Docker 服务器部署清晰。
- 后端控制力强。

注意：

- Cloudflare 免费路线适配成本更高。
- 需要自己维护更多部署细节。

## 推荐目录结构

如果采用一个全栈仓库，建议：

```text
JCC-Web-New/
  docs/
  app/
    src/
      pages/
      components/
      features/
        auth/
        streamers/
        fans/
        sessions/
        rankings/
        matches/
        tickets/
        screenshots/
        public/
        audit/
      lib/
        db/
        storage/
        auth/
        rules/
      styles/
    public/
    tests/
  migrations/
  scripts/
  README.md
  CHANGELOG.md
```

说明：

- `features/` 按业务模块拆分，避免旧项目那种单文件堆功能。
- `lib/rules/` 放票务、定榜、锁牌等纯业务规则，方便测试。
- `lib/storage/` 抽象本地文件和 R2。
- `migrations/` 保存数据库迁移。
- `docs/` 保留当前规划文档。

## 本地运行策略

本地 MVP：

- 使用 SQLite。
- 图片存到本地 `uploads/`。
- 管理员账号通过初始化脚本创建。
- 开发环境提供 seed 数据。

本地运行需要支持：

- 创建主播。
- 创建粉丝。
- 创建场次。
- 定榜、对局、存票、截图、公开榜完整流程。

## Cloudflare 部署策略

目标组合：

- 前端：Cloudflare Pages。
- API：Workers。
- 数据库：D1。
- 图片：R2。

注意：

- 避免高频实时轮询。
- 图片限制 5MB。
- 所有上传走后端校验。
- 本地和线上数据库迁移要保持一致。

## Docker 云服务器部署策略

目标组合：

- Web 服务容器。
- SQLite 或 PostgreSQL。
- 本地磁盘或对象存储保存图片。
- 反向代理和 HTTPS。

适合在 Cloudflare 不够用时切换。

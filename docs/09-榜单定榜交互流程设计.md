# 09-榜单定榜交互流程设计

整理时间：2026-06-20 10:52

## 背景判断

当前页面虽然已经从“创建定榜表单”改成了“本轮榜单池”，但业务模型仍然不够准确。

真实直播里，主播面对的不是一轮一轮孤立的定榜表单，而是一个持续变化的本场直播榜单。粉丝会在整场直播中不断现刷、取票、存票、补票、临时有事、让票、退票。定榜只是从当前榜单状态中冻结某一个时间点的结果。

因此下一步页面应从“本轮榜单池”继续调整为“本场榜单页面”。

核心结论：

```text
榜单页面 = 本场直播所有粉丝票务和上榜状态的实时工作台
定榜 = 对当前榜单启动倒计时并冻结一份结果
定榜记录 = 冻结时刻的榜单快照
名单确认 = 在冻结结果基础上人工调整上车名单
对局 = 名单确认后的游戏记录
```

## 关键概念

### 本场榜单

本场榜单是当前直播场次下的实时表格。

它记录本场所有和上榜有关的数据，包括：

- 粉丝。
- 本场总票。
- 现刷。
- 取票。
- 存票。
- 调整。
- 结算后存票余额预览。
- 新粉/老粉/管理/违规/拉黑。
- 本场状态：正常竞争、有事不来、待定、禁赛。
- 备注。

本场榜单不是冻结结果，可以随时改。

### 定榜

定榜不是一张从零开始录入的表。

定榜是一个动作：

```text
从本场榜单当前数据开始倒计时
  -> 倒计时期间榜单仍可更新
  -> 倒计时结束或手动冻结
  -> 生成本次定榜快照
```

### 定榜快照

定榜快照是冻结时刻的榜单结果。

它应该保存：

- 冻结时间。
- 定榜编号。
- 定榜名称。
- 冻结时的粉丝排序。
- 冻结时的票数明细。
- 冻结时的粉丝状态。
- 冻结时的备注。

定榜快照默认不再直接修改。如果确实需要修正，应通过“名单确认调整”或“管理员修正记录”处理，避免历史不可追溯。

### 名单确认

名单确认是在定榜快照基础上的人工裁决。

系统根据定榜规则生成推荐名单，但主播/管理员可以：

- 调整上车/待定。
- 标记有事不来。
- 补位老粉。
- 处理新粉数量变化。
- 备注特殊情况。
- 创建对局。

## 票数口径

用户当前确认的目标公式：

```text
本场总票 = 现刷 + 取票 - 存票
```

含义建议：

- `现刷`：本场直播收到的抖音礼物钻，默认参与上榜。
- `取票`：从历史存票余额中取出，用来参与本场上榜。
- `存票`：本场刷了但不参与上榜、转入长期余额的部分。
- `调整`：人工修正项，处理特殊情况。

更完整的计算建议：

```text
本场总票 = 现刷 + 取票 - 存票 + 调整
```

结算后存票余额预览：

```text
结算后余额 = 当前长期余额 - 取票 + 存票 + 回退 + 余额调整
```

其中：

- 如果取票后没有上车，可以在名单确认或结算时标记未使用，并作为 `回退` 加回余额。
- 现刷如果全部参与上榜，不进入长期余额。
- 现刷如果其中一部分要存票，应填入 `存票`，从本场总票里扣掉。

## 需要确认的口径

这里有一个必须确认的命名问题：

如果 `现刷` 表示“本场收到的全部礼物钻”，则公式 `总票 = 现刷 + 取票 - 存票 + 调整` 是合理的。

如果 `现刷` 表示“本场明确用于上榜的礼物钻”，则不应该再减存票。

建议页面上把字段命名成：

- `礼物钻`：本场实际收到的礼物钻。
- `存票`：从礼物钻中转入长期余额、不参与上榜的部分。
- `取票`：从历史余额中取出参与上榜。
- `调整`：人工修正。
- `本场总票`：礼物钻 + 取票 - 存票 + 调整。

这样比“现刷”更不容易歧义。

如果你仍想保留“现刷”这个词，建议在页面上写成：

```text
现刷礼物
```

## 页面总体结构

当前场次工作台中的 `定榜` 标签，建议改名为：

```text
榜单/定榜
```

页面分为四个区域：

```text
顶部：本场榜单控制条
中部：本场实时榜单表格
右侧或下方：快速编辑区
底部：定榜记录和冻结名单
```

在桌面端：

```text
左/中：榜单表格
右：定榜控制 + 快速编辑
下：定榜记录 / 冻结名单
```

在手机端：

```text
顶部：控制条
中部：榜单表格，横向滚动或卡片压缩
下方：快速编辑
底部：定榜记录
```

## 顶部控制条

顶部显示当前场次和榜单状态。

内容：

- 当前场次名称。
- 主播名称。
- 当前场次状态。
- 当前榜单人数。
- 正常竞争人数。
- 新粉人数。
- 有事不来人数。
- 待冻结/倒计时/已冻结状态。

操作：

- 添加粉丝到榜单。
- 开始定榜。
- 设置倒计时。
- 手动冻结。
- 查看定榜记录。

## 本场实时榜单表格

表格是核心。

建议列：

| 列 | 说明 |
| --- | --- |
| 排名 | 根据本场总票实时计算 |
| 粉丝 | 显示粉丝名，可带新粉/老粉/管理/违规/拉黑标签 |
| 礼物钻/现刷 | 本场实际收到的礼物钻 |
| 取票 | 从历史存票中取出用于本场上榜 |
| 存票 | 本场礼物中转入长期余额、不参与上榜的部分 |
| 调整 | 人工加减 |
| 本场总票 | 礼物钻 + 取票 - 存票 + 调整 |
| 当前余额 | 操作前长期余额 |
| 结算后余额预览 | 当前余额 - 取票 + 存票 + 回退 + 余额调整 |
| 状态 | 正常、新粉、有事不来、待定、禁赛 |
| 同票顺序 | 同总票时人工排序 |
| 备注 | 本场备注 |
| 操作 | 编辑、标记有事、移出本场榜单 |

### 表格编辑方式

建议支持两种编辑方式。

#### 方式 A：表格内直接编辑

适合电脑端。

可直接修改：

- 礼物钻。
- 取票。
- 存票。
- 调整。
- 状态。
- 同票顺序。
- 备注。

修改后自动保存或点击保存。

建议第一版使用“行内编辑 + 保存按钮”，不要做完全自动保存，避免直播中误触。

#### 方式 B：下方/右侧快速编辑区

适合手机和平板。

流程：

```text
选择粉丝
  -> 填礼物钻/取票/存票/调整/备注
  -> 保存
  -> 表格刷新排序
```

快速编辑区还应该支持：

- 搜索粉丝。
- 新增粉丝。
- 标记新粉。
- 标记有事不来。
- 标记恢复正常。

## 排序规则

正常竞争榜排序：

```text
本场总票降序
  -> 同票顺序升序
  -> 更新时间或首次上榜时间兜底
```

状态特殊处理：

- 拉黑/禁赛：不进入正常竞争榜，单独区域提醒。
- 有事不来：不进入正常竞争榜，单独区域显示。
- 待定：可以留在正常榜或单独显示，建议第一版先留在正常榜并标记。
- 新粉：仍参与正常榜，但标签突出显示。

分区建议：

```text
正常竞争榜
本场新粉
有事不来的老粉/粉丝
禁赛/拉黑
```

这里的 `本场新粉` 可以不是独立榜单，而是在正常竞争榜顶部/旁边显示统计；如果新粉较重要，可以加一个筛选按钮。

## 定榜流程

### 1. 开始定榜

用户点击：

```text
开始定榜
```

弹窗或面板显示：

- 定榜编号。
- 定榜名称。
- 倒计时时长，默认 3 分钟。
- 定榜规则：定榜七 / 定榜五 / 手动。
- 是否立即开始倒计时。

默认名称：

```text
第 N 次定榜
```

如果本场有手动编号调整，则按调整后的编号继续往后排。

### 2. 倒计时

倒计时默认：

```text
3:00
```

当前系统出现 `3:04` 的原因大概率是：

- 后端按 `Date.now() + 180s` 生成结束时间。
- 前端拿到接口响应后才开始渲染。
- 本地时间/请求延迟/定时器刷新造成显示误差。

建议改为：

```text
前端点击时先显示 3:00
后端返回 startedAt / endsAt
前端用 endsAt 精准校准
显示时向下取整，不向上取整
```

当前应避免 `Math.ceil` 导致刚开始显示 `3:01` 或更大。

倒计时期间：

- 榜单仍然可以录入和修改。
- 页面明显显示“倒计时中”。
- 可手动冻结。
- 可取消倒计时，回到编辑状态。

### 3. 冻结

冻结触发方式：

- 倒计时结束后手动点击冻结。
- 倒计时结束后系统提示冻结。
- 主播提前手动冻结。

第一版建议：

```text
倒计时结束后不自动冻结，只高亮提示“倒计时结束，请冻结榜单”
```

原因：

直播现场可能最后几秒还在补票，完全自动冻结容易和现场口头规则冲突。

冻结时保存：

- 当前排序。
- 所有粉丝票数。
- 所有状态。
- 所有备注。
- 冻结时间。
- 操作者。

冻结后生成：

```text
本次冻结名单
```

## 冻结名单

冻结名单显示在榜单下方。

内容：

- 本次定榜编号。
- 冻结时间。
- 定榜规则。
- 正常竞争排序。
- 有事不来。
- 禁赛/拉黑。
- 推荐上车名单。
- 待定名单。

冻结名单不是最终对局名单。

下一步是：

```text
进入名单确认
```

## 名单确认

名单确认区域应支持：

- 显示 8 个席位。
- 主播固定席位。
- 推荐上车粉丝。
- 待定池。
- 有事不来的粉丝不占席位。
- 禁赛/拉黑不能进入席位。
- 手动把待定粉丝拖入/移出。
- 记录游戏名备注。
- 确认后创建对局。

定榜七/定榜五规则应只影响推荐，不应禁止人工调整。

## 定榜记录

页面应有定榜记录区域。

每条记录显示：

- 第几次定榜。
- 定榜名称。
- 规则。
- 冻结时间。
- 是否已创建对局。
- 操作者。
- 查看详情。
- 作废/修正。

查看详情时显示冻结时完整快照。

## 定榜编号规则

默认自动编号：

```text
第 1 次定榜
第 2 次定榜
第 3 次定榜
```

支持手动调整编号。

场景：

```text
中间有一局对局忘记记录
用户手动把当前定榜编号改成第 5 次
后续自动从第 6 次继续
```

建议规则：

- `round_no` 可以手动编辑。
- 同一场次下 `round_no` 不建议重复。
- 如果用户手动设置为 5，下一次自动编号取 `max(round_no) + 1`。
- 如果确实需要重复，必须给出提醒。

定榜名称默认：

```text
第 {round_no} 次定榜
```

支持手动改名：

```text
第 5 次定榜（补记）
晚上场第 2 把
```

## 数据模型建议

当前已有：

- `ranking_snapshots`
- `ranking_entries`
- `ticket_ledgers`

但如果改成“本场榜单页面”，建议新增或调整一个概念：

### live_session_board_entries

本场实时榜单表。

建议字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `session_id` | 场次 |
| `fan_id` | 粉丝 |
| `gift_diamonds` | 本场礼物钻/现刷 |
| `ticket_used` | 本场取票 |
| `ticket_deposit` | 本场存票 |
| `manual_adjustment` | 调整 |
| `competition_score` | 本场总票 |
| `balance_preview` | 结算后余额预览，可实时算不一定入库 |
| `status` | normal / away / pending / blocked |
| `tie_order` | 同票顺序 |
| `note` | 本场备注 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### ranking_snapshots

定榜快照表。

用于冻结时保存一次结果。

建议字段：

- `session_id`
- `round_no`
- `title`
- `style`
- `status`
- `countdown_seconds`
- `countdown_started_at`
- `countdown_ends_at`
- `frozen_at`
- `created_by`
- `note`

### ranking_entries

冻结快照条目表。

冻结时从 `live_session_board_entries` 复制。

这样能解决：

- 本场榜单持续变化。
- 每次定榜能保存历史快照。
- 后续继续改榜单不会影响历史定榜记录。

## 是否需要新增表

建议新增 `live_session_board_entries`。

原因：

当前用 `ranking_entries` 同时承担“实时榜单”和“冻结快照”，概念会混乱。

如果继续复用 `ranking_entries`：

- 每次定榜之前的数据到底属于当前榜单还是快照不清楚。
- 冻结后继续录入下一轮会和历史结果混在一起。
- 本场榜单不能自然跨多次定榜延续。

更合理结构：

```text
live_session_board_entries = 本场实时榜单
ranking_snapshots = 每次定榜动作
ranking_entries = 每次冻结快照
```

## 操作流程总图

```text
进入当前场次
  -> 打开榜单/定榜
  -> 本场榜单表格持续录入
      -> 礼物钻
      -> 取票
      -> 存票
      -> 调整
      -> 状态
      -> 备注
  -> 实时计算本场总票和余额预览
  -> 点击开始定榜
      -> 设置编号/名称/规则/倒计时
      -> 倒计时中继续允许更新榜单
  -> 倒计时结束或手动冻结
      -> 复制当前榜单为定榜快照
      -> 生成冻结名单
  -> 名单确认
      -> 调整上车/待定/补位
      -> 确认 8 人席位
  -> 创建对局
  -> 进入锁牌
```

## 第一版实现建议

不要一次做太复杂。建议分三步：

### 第一步：重建榜单页面

- 新增本场榜单表格。
- 支持添加粉丝。
- 支持修改礼物钻、取票、存票、调整、状态、备注。
- 实时计算总票和余额预览。
- 支持正常竞争、有事不来、禁赛分区。

### 第二步：定榜冻结

- 开始定榜。
- 设置倒计时，默认 3 分钟。
- 修正倒计时显示从 `3:00` 开始。
- 手动冻结。
- 冻结时生成定榜快照。
- 查看定榜记录。

### 第三步：名单确认和创建对局

- 从冻结快照生成推荐名单。
- 支持手动调整。
- 创建对局。

## 暂不做

第一版暂不做：

- 自动读取抖音榜单。
- OCR 识别截图。
- 拖拽排序。
- 多人实时协同编辑冲突处理。
- 自动冻结后直接创建对局。
- 高级统计图表。

## 当前需要你确认的问题

1. 页面字段是否使用 `礼物钻` 代替 `现刷`？

建议使用 `礼物钻`，公式为：

```text
本场总票 = 礼物钻 + 取票 - 存票 + 调整
```

2. 倒计时结束后是否自动冻结？

建议第一版不自动冻结，而是提示“倒计时结束，请冻结榜单”，由主播手动点。

3. `本场新粉` 是否需要单独区域，还是只在正常榜里用标签和筛选显示？

建议第一版用标签和筛选，不单独拆榜，避免同一个粉丝在多个区域重复出现。

4. 本场榜单是否允许把粉丝移出？

建议允许“移出本场榜单”，但不删除粉丝资料，不删除票务流水。

5. 存票是否一定来自本场礼物钻？

如果存票一定是“本场刷了但要存起来”，则 `总票 = 礼物钻 + 取票 - 存票 + 调整` 成立。

如果存票也可能是单纯补录历史存票，不应该影响本场总票，需要在页面上区分“本场存票”和“历史补录存票”。

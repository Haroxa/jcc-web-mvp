# AGENTS.md

本文件是 Codex 在本项目中的工作规则。开始修改前请先阅读本文。

## 项目概况

- 项目名称：JCC Web New
- 项目目标：为金铲铲水友赛主播提供可部署的直播辅助工具，覆盖主播端、管理端和游客公开页。
- 项目类型：可部署 Web 应用。
- 技术路线：TypeScript + React + Drizzle + SQLite，预留 Cloudflare Workers / D1 / R2 适配。
- 当前阶段：已完成账号、主播、粉丝、场次、票务和定榜基础，正在重组前端信息架构。
- UI 方向：朴素后台风格，优先功能完整、操作清楚、数据可靠；主播端围绕当前场次做直播操作台，管理端保留后台管理和代操作能力。

## 当前关键业务范围

- 主播端：直播场次、定榜、推荐名单、锁牌、存票、截图、结算。
- 管理端：主播账号、代操作、资料管理、日志、异常检查。
- 游客端：公开存票榜和公开内容查看。
- 粉丝不登录、不注册。
- 管理员和主播登录。
- 管理员可以代主播操作。

## 常用命令

在项目根目录 `JCC-Web-MVP` 下执行：

```bash
npm install
npm run dev
npm run typecheck
npm run build
npm run db:generate
```

Cloudflare 相关命令：

```bash
npm run worker:dev
npm run worker:deploy
npm run db:migrate:local
npm run db:migrate:remote
```

注意：Cloudflare 命令依赖 `npx wrangler`，需要网络可用和 Cloudflare 配置完成。

## 工作规则

- 修改前先确认影响范围，尤其是数据库 schema、迁移、票务计算和公开字段。
- 不要直接复用旧项目源码；旧项目只作为业务经验参考。
- 保持模块边界清晰，不把所有逻辑堆到一个文件。
- 主播端页面优先围绕当前场次组织，不把直播中高频操作拆成多个孤立 CRUD 页面。
- 票务、锁牌、名单生成等核心规则优先写成可测试的纯函数。
- 票务不直接硬改历史记录，优先用流水、修正、作废和日志。
- 截图默认不公开，公开由显式字段控制。
- 游客页只能展示公开字段，也只能按公开字段筛选。
- 数据库变更必须同步更新 `docs/06-数据库草案.md` 或迁移说明。
- 重要功能进展必须更新 `CHANGELOG.md` 和 `docs/项目状态.md`。
- 新问题或踩坑必须记录到 `docs/问题汇总与解决.md`。

## 代码和文档拆分规则

为避免后续继续把逻辑堆回大文件，新增或修改功能时必须遵守以下边界：

### 文件长度阈值

- TypeScript / TSX 文件超过 500 行时，必须优先拆分后再继续新增功能。
- TypeScript / TSX 文件达到 400 行以上时，新增功能前必须先评估是否应该拆成子组件、路由模块、服务函数或工具模块。
- 不要为了少量复用强行抽象；但如果一个文件同时承担入口、状态管理、表单、列表渲染、业务请求和数据转换，应优先拆分。
- 新增代码时不允许把入口文件重新堆大：`src/App.tsx` 只负责登录、导航和视图切换；`worker/index.ts` 只负责 Hono/CORS、健康检查和路由注册；`worker/shared.ts` 只作为统一出口。

### 前端模块边界

- 页面级组件放在 `src/views/`，按业务页面命名，例如 `FanManager.tsx`、`TicketManager.tsx`、`CurrentSessionWorkspace.tsx`。
- 页面内纯展示分组、表格、卡片或重复区块应拆成同目录子组件文件，例如 `RankingGroups.tsx`。
- 共享类型放在 `src/types.ts`；共享常量放在 `src/constants.ts`；请求封装放在 `src/api/client.ts`；格式化和标签映射放在 `src/utils/`。
- 新增直播现场高频操作时，优先接入当前场次工作台或对应业务页面，不要在 `App.tsx` 内直接实现页面细节。

### 后端模块边界

- 所有 API 路由按业务域放在 `worker/routes/`，并通过 `registerXxxRoutes(app)` 在 `worker/index.ts` 注册。
- Worker 共享逻辑按用途放在 `worker/lib/`：
  - `types.ts`：Bindings、数据库行类型和跨模块类型。
  - `core.ts`：时间、ID、密码、错误响应等基础工具。
  - `accounts.ts`：账号、会话、审计、主播可访问空间。
  - `fans.ts`：粉丝状态解析、归一化和转换。
  - `sessions.ts`：场次状态、类型和转换。
  - `tickets.ts`：票务类型、余额影响和余额刷新。
  - `rankings.ts`：本场榜单、定榜、冻结和推荐相关逻辑。
- 新增后端能力时先判断属于哪个业务域；只有真正跨域复用的函数才进入 `worker/lib/`。
- Worker 代码必须继续使用 `context.env` 访问 D1/R2 等绑定，不要引入模块级请求状态。

### 文档长度治理

- 单个文档超过 800 行时，应评估是否拆为索引文档 + 分主题文档。
- `CHANGELOG.md` 可保留最新变更和索引；历史记录过长时按月份或阶段归档到 `docs/changelog/`。
- 设计文档超过 800 行时，优先按页面或业务流程拆分，例如当前场次、榜单定榜、结算、管理端、游客页。
- 问题记录超过 800 行时，优先按部署/数据/权限/业务流程或 P0/P1 分组归档。
- 拆分文档时必须保留原入口文档作为目录或索引，不要让历史信息失去入口。

## 文档维护规则

所有需要持续维护的文档，更新时间统一使用 `yyyy-MM-dd HH:mm`，并按真实变更同步更新。不要只改代码不改状态，也不要只写变更记录不更新风险或计划。

| 文档 | 更新频率 | 必须更新的情况 |
| --- | --- | --- |
| `CHANGELOG.md` | 每次形成可提交变更时 | 文件、配置、代码、部署、文档结构发生明确变化 |
| `docs/项目状态.md` | 每次阶段状态变化时 | 功能从待做变成进行中/已完成，部署状态变化，验证项变化 |
| `docs/项目功能执行计划清单.md` | 每次功能推进时 | 新增/删除功能，优先级变化，功能状态变化，下一步顺序变化 |
| `docs/问题汇总与解决.md` | 每次遇到问题或完成验证时 | 依赖、构建、部署、运行、数据、权限、环境问题出现或解决 |
| `docs/风险清单.md` | 每次发现或降低风险时 | 发现新的数据/权限/部署/使用风险，已有风险状态变化 |
| `docs/决策记录.md` | 每次做出长期决策时 | 技术栈、部署方式、权限模型、数据模型、业务规则确认 |
| `docs/06-数据库草案.md` | 每次数据结构变化时 | 表、字段、索引、枚举、迁移策略变化 |
| `README.md` | 对外启动/部署方式变化时 | 命令、环境变量、部署步骤、项目说明变化 |

### 文档结构要求

- `docs/问题汇总与解决.md` 必须先有问题概览，再有时间线，最后是详细问题记录。
- `docs/风险清单.md` 必须先有风险概览，再列当前风险、发布前检查和暂不处理项。
- `docs/项目功能执行计划清单.md` 必须先有进度概览，再列功能计划和下一步顺序。
- 已解决问题不要删除，状态改为已解决，并保留验证方式。
- 已降低但仍可能复发的风险不要删除，状态改为已缓解或持续关注。
- 功能计划不要只写“待做”，应尽量标注 `待做`、`进行中`、`已完成初版`、`已验证`、`暂缓`。

## Git 提交规则

- 每完成一组明确任务后，应创建一次 Git 提交，方便后续追溯和回滚。
- 提交前先检查 `git status --short --branch`，确认没有误加入 `node_modules/`、`dist/`、密钥、真实数据或临时文件。
- 涉及代码时，优先运行 `npm run typecheck` 和 `npm run build`。
- 涉及文档时，至少确认相关文档时间、变更记录和状态说明已同步。
- 不把半成品调试过程拆成大量零碎提交；一个提交应该对应一个清晰目标。
- 提交信息使用中文短句，说明本次变更目标，例如 `初始化新项目骨架和项目文档`。
- 如果用户明确要求暂不提交，则只保留工作区改动并说明当前状态。

## 不要默认修改

- 不要删除用户数据、截图、迁移文件和配置文件。
- 不要提交真实账号、密码、token、Cloudflare id 或密钥。
- 不要默认改 npm/pip 全局代理配置，除非用户明确要求。
- 不要默认切换技术栈。
- 不要引入大型 UI 框架或复杂视觉方案，除非确实需要。
- 不要把旧项目的本地存储模型作为新项目实现方式。
